# 对话式移动端前端 PRD v1.1

## 1. 文档信息

- 文档名称：对话式移动端前端 PRD v1.1
- 版本：v1.1
- 当前方案：消息体验与状态感知升级方案
- 面向对象：产品、前端、后端、设计、测试
- 适用范围：手机浏览器 H5 前端 v1 基础上的体验增强版本

## 2. 版本背景

PRD v1 已经完成移动端一期的基础闭环：

- 用户登录后直接进入当前会话页
- 用户可在同一页面完成分析发起、处理中查看、结果阅读与继续追问
- 新建会话、历史会话切换、账户操作统一收纳到左侧抽屉

在 v1 实际使用和持续迭代过程中，当前消息流仍存在三类明显体验缺口：

1. 用户对“是谁在回答自己”感知不稳定，系统回复、结果卡片、错误提示的角色认知不统一
2. 对话消息的元信息占用空间偏大，消息流密度不够，移动端连续阅读时显得松散
3. 长时间分析过程中，用户虽然能看到状态反馈，但还不能明显感受到“一个投资团队正在持续为自己工作”

因此，v1.1 不改变整体产品结构，不新增复杂页面体系，而是聚焦对消息体验和任务状态体验做一次明确升级。

## 3. 产品定位

v1.1 延续 v1 的产品定位：

**一个面向手机浏览器的轻量投研对话工作台。**

但在消息体验层面，v1.1 会进一步强化两个感知：

1. 用户始终在和同一个前台角色进行交流
2. 分析过程中，后台投资团队正在持续为自己推进工作

v1.1 的目标不是增加更多功能页面，而是在现有单会话工作区中，让消息流更统一、更紧凑、更有工作感。

## 4. v1.1 目标与非目标

### 4.1 v1.1 目标

- 统一面向用户的系统发言角色，降低消息来源认知成本
- 提升消息流密度，让移动端阅读更紧凑、更接近聊天产品
- 强化任务处理中“投资团队正在工作”的感知，降低长任务等待的不确定性
- 在不破坏 v1 已有信息架构的前提下，完成一次消息体验增强

### 4.2 v1.1 非目标

- 不改变登录、抽屉、历史会话、当前会话页的主结构
- 不新增复杂后台页、任务看板或系统监控面板
- 不展示底层 Agent 名称、模型名称、内部服务名或技术阶段码
- 不引入复杂百分比进度条或技术监控式进度体系
- 不重做现有消息存储协议和历史消息迁移

## 5. v1.1 核心设计原则

### 5.1 对用户的系统发言人格统一

面向用户的系统消息应以统一角色出现，减少“系统 / 助手 / 卡片 / 错误提示”多主体并存造成的认知割裂。

### 5.2 保持消息流是聊天而不是文档

消息元信息与正文应更紧凑，优先提升聊天感和连续阅读感，不将对话页做成文档页或监控页。

### 5.3 状态反馈强调“团队在工作”

任务状态消息应让用户感知“哪个团队/成员正在为自己推进分析”，而不是仅感知系统还未卡死。

### 5.4 结果与状态分工清晰

统一前台发言角色与后台工作进度反馈应分工明确：

- 前台角色负责对用户说话
- 任务状态负责表达团队推进过程

### 5.5 以体验升级为主，不扩大产品边界

v1.1 是对消息体验和状态反馈的升级，不新增新的复杂业务闭环，不改变 v1 的主路径。

## 6. v1.1 核心方案概述

### 6.1 新增统一系统角色“小I”

v1.1 引入统一系统角色：

- `小I`

`小I` 是用户在移动端会话中的统一前台对话角色。凡是需要直接向用户表达内容、确认信息、解释结果、提示异常的消息，都由 `小I` 发出。

### 6.2 消息元信息紧凑化

v1.1 将普通对话消息中的角色与时间戳放入同一行展示，减少垂直占用，提升消息流密度。

### 6.3 任务状态升级为“投资团队工作进度反馈”

v1.1 中的 `task_status` 不再只表达系统状态，而是表达：

- 某个团队 / 某个成员
- 正在为用户做什么
- 当前阶段是否已完成

## 7. v1.1 功能范围

### 7.1 必做功能

### F6. 小I 对话角色统一化

#### 目标

让用户在会话中始终感知自己是在和同一个前台角色交流，而不是面对多种分裂的系统口吻。

#### 功能点

- 新增系统角色名称：`小I`
- 面向用户的系统消息统一展示为 `小I`
- 以下消息类型纳入 `小I` 发言范围：
  - `text`
  - `ticker_resolution`
  - `insight_reply`
  - `summary_card`
  - `error`

#### 产品规则

- `小I` 是统一前台角色，不等同于所有系统输出
- `task_status` 不作为 `小I` 发言，而是单独承担团队工作进度反馈职责
- `小I` 的引入优先体现在前端展示层，不要求本期重构全部底层消息 role 存储

#### 交互要求

- 用户消息展示角色为：`你`
- 系统对话类消息展示角色为：`小I`
- `summary_card` 和 `error` 也要延续 `小I` 的统一口吻，不再呈现割裂的系统角色体验

#### 一期边界

- v1.1 不支持用户自定义系统角色名称
- v1.1 不支持多个系统角色并存
- v1.1 不要求重做历史消息存量数据，只要求新增与当前渲染链路保持统一展示

#### 验收标准

- 用户在会话页能稳定感知系统发言角色为 `小I`
- `text / ticker_resolution / insight_reply / summary_card / error` 五类消息在前端展示层角色统一
- `task_status` 继续保持独立状态反馈定位，不与 `小I` 混淆

### F7. 消息元信息紧凑化

#### 目标

提升消息流密度，减少单条消息的垂直占用，让移动端聊天阅读更紧凑、更流畅。

#### 功能点

- 对话消息中的角色和时间戳放在同一行展示
- 普通对话消息元信息收敛为更轻的一层
- 消息正文与元信息之间的关系更清晰

#### 适用范围

本功能主要适用于以下互动型消息：

- `text`
- `ticker_resolution`
- `insight_reply`
- `error`

对 `summary_card` 允许保留其结果卡片属性，但应尽量与普通消息的元信息结构保持一致性。

#### 交互要求

- 元信息行采用：
  - 左侧角色
  - 右侧时间戳
- 时间戳继续保持轻量展示，不喧宾夺主
- 单条消息在视觉上更接近聊天产品，不演化成卡片式文档段落

#### 一期边界

- v1.1 不要求引入“刚刚 / 今天 / 昨天”之类的新时间体系
- v1.1 不要求所有消息类型完全共用同一模板
- `task_status` 不强制套用这一元信息布局

#### 验收标准

- 普通对话消息中，角色与时间戳处于同一行
- 单条消息高度相较 v1 有明显收敛
- 消息流整体阅读密度提升，但不破坏聊天感
- 用户仍可清楚区分“你”和“小I”的发言

### F8. 投资团队工作进度反馈增强

#### 目标

让用户在长任务处理中持续感知“投资团队正在为自己工作”，而不是只看到抽象的系统处理中提示。

#### 功能点

- 活跃 `task_status` 增加轻量转圈动效
- 已完成 `task_status` 增加完成标记
- 已用时间、预计剩余时间的提示更突出
- `task_status` 文案从系统状态升级为团队/成员工作进度反馈

#### 文案原则

`task_status` 应统一使用“谁在做什么”的表达，而不是技术状态表达。

建议采用如下结构：

- `<团队/成员> 正在 <动作>`
- `<团队/成员> 已完成 <动作>`

示例方向：

- `新闻分析师正在整理近期关键事件`
- `基本面分析师正在梳理盈利与估值`
- `风险分析师正在评估下行风险`
- `投资总监正在汇总结论`

#### 交互要求

- 活跃态：
  - 左侧使用轻量动态标记
  - 主文案表达团队/成员正在做什么
  - 已用、预计剩余时间要比当前更容易被看到
- 完成态：
  - 停止动效
  - 使用完成标记
  - 明确表达该阶段已完成

#### 产品行为约束

- `task_status` 仍是状态反馈，不是聊天回复正文
- `task_status` 不承担结果卡片职责
- `task_status` 不暴露底层 Agent 名、模型名、服务名、阶段 ID 等技术细节
- 即使阶段没有切换，也应通过时间变化与轻量动态状态让用户感知任务仍在继续

#### 一期边界

- v1.1 不引入复杂百分比进度条
- v1.1 不展示底层多 Agent 编排图
- v1.1 不做技术监控面板式任务看板
- v1.1 只要求用户感知“投资团队正在持续为自己工作”

#### 验收标准

- 用户在分析进行中，能持续感知有团队/成员正在推进工作
- 活跃态 `task_status` 具有明确主体与动作表达
- 完成态 `task_status` 有完成标记，且不再保留进行中动效
- 已用时间与预计剩余时间比 v1 更突出，但不压过主状态文案
- 用户在长任务处理中不会因页面静止而误以为系统挂起
- 全部状态文案不暴露内部技术实现细节

## 8. 与 v1 的关系

v1.1 是对 v1 的增量升级，不改变以下基础结构：

- 登录后默认进入当前会话页
- 左侧抽屉承载导航、历史会话与账户菜单
- 当前会话页仍是唯一主工作区
- 会话式 ticker 识别与确认流程不变
- 分析结果仍通过消息流承接

v1.1 的核心变化只发生在：

- 消息角色统一
- 消息元信息布局
- 任务状态表达方式

## 9. 风险与注意事项

### 9.1 角色统一带来的边界混淆风险

若不明确区分 `小I` 与 `task_status` 的职责，容易让用户误以为所有状态消息都是聊天回答，从而削弱“团队正在工作”的感知。

### 9.2 过度人格化风险

如果 `task_status` 文案过度拟人化或戏剧化，可能削弱专业投研产品的可信度。因此文案应保持专业、轻量、有工作感，而不是表演感。

### 9.3 过度技术化风险

若将内部技术阶段直接映射到用户界面，会破坏产品的专业感与可理解性。v1.1 必须优先让用户理解“谁在为自己做什么”，而不是系统内部如何调度。

## 10. 验收结论

v1.1 上线后，移动端对话体验应达到以下整体结果：

- 用户能够明确分辨“你”和“小I”
- 系统类对话输出具备统一的前台角色体验
- 消息流更紧凑，移动端阅读负担更低
- 长分析过程中，用户能明确感知投资团队正在持续推进工作
- 整体体验比 v1 更像一个有统一人格、有持续反馈的投研对话工作台
