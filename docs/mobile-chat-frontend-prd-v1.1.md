# 对话式移动端前端需求说明书 v1.1

## 1. 文档信息

- 文档名称：对话式移动端前端需求说明书 v1.1
- 版本：v1.1
- 当前方案：消息体验与状态感知升级方案
- 文档类型：需求说明书
- 面向对象：产品、前端、后端、设计、测试
- 适用范围：手机浏览器 H5 前端 v1 基础上的体验增强版本

## 2. 背景与问题定义

### 2.1 当前版本基础能力

PRD v1 已完成移动端一期基础闭环：

- 用户登录后直接进入当前会话页
- 用户可在同一页面完成分析发起、处理中查看、结果阅读与继续追问
- 新建会话、历史会话切换、账户操作统一收纳到左侧抽屉

### 2.2 当前问题

在 v1 实际使用和持续迭代过程中，当前消息流存在以下问题：

1. 用户对“是谁在回答自己”感知不稳定，系统回复、结果卡片、错误提示的角色认知不统一。
2. 对话消息元信息占用空间偏大，消息流密度不足，移动端连续阅读时显得松散。
3. 长时间分析过程中，用户虽然能看到状态反馈，但还不能明显感受到“一个投资团队正在持续为自己工作”。

### 2.3 本次改版定位

v1.1 不改变整体产品结构，不新增复杂页面体系，而是在现有单会话工作区内完成以下体验升级：

- 统一前台对话角色
- 提升消息流阅读密度
- 强化任务处理中“投资团队正在工作”的感知

## 3. 产品定位

v1.1 延续 v1 的产品定位：

**一个面向手机浏览器的轻量投研对话工作台。**

本次升级强化两个核心感知：

1. 用户始终在和同一个前台角色进行交流。
2. 分析过程中，后台投资团队正在持续为用户推进工作。

## 4. 目标与非目标

### 4.1 产品目标

- 统一面向用户的系统发言角色，降低消息来源认知成本
- 提升消息流密度，让移动端阅读更紧凑、更接近聊天产品
- 强化任务处理中“投资团队正在工作”的感知，降低长任务等待的不确定性
- 在不破坏 v1 既有信息架构的前提下完成体验增强

### 4.2 非目标

- 不改变登录、抽屉、历史会话、当前会话页的主结构
- 不新增复杂后台页、任务看板或系统监控面板
- 不展示底层 Agent 名称、模型名称、内部服务名或技术阶段码
- 不引入复杂百分比进度条或技术监控式进度体系
- 不重做现有消息存储协议和历史消息迁移
- 不支持用户自定义系统角色名称
- 不支持多个系统角色并存

## 5. 适用范围与影响范围

### 5.1 适用范围

本需求适用于移动端 H5 当前会话页的消息流区域，包括：

- 用户消息
- 系统消息
- 结果卡片
- 错误消息
- 任务状态消息

### 5.2 不受影响范围

以下能力保持 v1 不变：

- 登录流程
- 左侧抽屉结构
- 历史会话切换逻辑
- 当前会话页为唯一主工作区的结构
- 会话式 ticker 识别与确认主流程
- 分析结果通过消息流承接的总体方式

## 6. 角色与术语定义

### 6.1 前台角色

- `你`：用户在消息流中的展示角色。
- `小I`：系统统一前台对话角色。凡是直接向用户表达内容、确认信息、解释结果、提示异常的消息，统一展示为 `小I`。

### 6.2 状态角色

- `task_status`：任务状态反馈，不属于 `小I` 发言，不承担聊天回复正文职责，用于表达投资团队推进过程。

### 6.3 术语说明

- 消息元信息：消息顶部或消息首行中的角色名、时间戳等信息。
- 活跃态：任务仍在处理中，状态会持续变化。
- 完成态：当前阶段已完成，不再展示进行中动效。
- 异常态：任务由于超时、失败、网络异常等原因未能正常推进。

## 7. 需求原则

### 7.1 统一前台人格

面向用户的系统消息应以统一角色出现，减少“系统 / 助手 / 卡片 / 错误提示”并存造成的认知割裂。

### 7.2 保持聊天感

消息元信息与正文应更紧凑，优先提升聊天感和连续阅读感，不将对话页演化为文档页或监控页。

### 7.3 状态反馈体现工作推进

任务状态消息应让用户感知“哪个团队或成员正在为自己推进分析”，而不是仅感知系统未卡死。

### 7.4 结果与状态分工清晰

- 前台角色负责直接对用户说话
- 状态消息负责表达任务推进过程

### 7.5 以体验升级为主

本次仅对消息体验与状态反馈升级，不扩大产品边界，不新增新的复杂业务闭环。

## 8. 功能需求总览

本次版本包含 3 项必做需求：

- F6：小I 对话角色统一化
- F7：消息元信息紧凑化
- F8：投资团队工作进度反馈增强

## 9. 详细功能需求

### 9.1 F6 小I 对话角色统一化

#### 9.1.1 需求目标

让用户在会话中始终感知自己在和同一个前台角色交流，而不是面对多种分裂的系统口吻。

#### 9.1.2 功能规则

- 新增系统角色名称：`小I`
- 面向用户的系统消息统一展示为 `小I`
- `小I` 的引入优先体现在前端展示层，不要求本期重构全部底层消息 role 存储
- `task_status` 不作为 `小I` 发言，而是单独承担团队工作进度反馈职责

#### 9.1.3 消息类型映射规则

| 消息类型 | 前端展示角色 | 展示模板 | 是否纳入小I | 说明 |
| --- | --- | --- | --- | --- |
| `user_text` | `你` | 普通对话消息 | 否 | 用户输入内容 |
| `text` | `小I` | 普通对话消息 | 是 | 系统普通文本回复 |
| `ticker_resolution` | `小I` | 普通对话消息 | 是 | ticker 识别与确认相关消息 |
| `insight_reply` | `小I` | 普通对话消息 | 是 | 投研分析回复 |
| `summary_card` | `小I` | 结果卡片 | 是 | 允许保留卡片属性，但角色统一 |
| `error` | `小I` | 错误消息 | 是 | 以统一口吻表达异常 |
| `task_status` | 无前台角色或状态标签 | 状态消息 | 否 | 独立承担任务进度反馈 |

#### 9.1.4 交互要求

- 用户消息展示角色固定为 `你`
- 系统对话类消息展示角色固定为 `小I`
- `summary_card` 和 `error` 须延续 `小I` 的统一口吻
- 历史消息在前端回放时，如命中上述消息类型，也应按当前规则统一显示为 `小I`

#### 9.1.5 边界约束

- 不支持用户自定义系统角色名称
- 不支持多个系统角色并存
- 不通过本需求修改历史消息底层存储结构

### 9.2 F7 消息元信息紧凑化

#### 9.2.1 需求目标

提升消息流密度，减少单条消息垂直占用，让移动端聊天阅读更紧凑、更流畅。

#### 9.2.2 适用范围

本功能适用于以下互动型消息：

- `text`
- `ticker_resolution`
- `insight_reply`
- `error`

`summary_card` 允许保留结果卡片属性，但应尽量与普通消息的元信息结构保持一致。

#### 9.2.3 布局规则

- 对话消息中的角色和时间戳放在同一行展示
- 元信息行采用“左侧角色、右侧时间戳”的布局
- 时间戳保持轻量展示，不喧宾夺主
- 消息正文与元信息应形成清晰层级，但不得造成明显垂直浪费
- `task_status` 不强制套用该元信息模板

#### 9.2.4 视觉要求

- 普通消息视觉上应更接近聊天产品，不演化成卡片式文档段落
- 消息角色名应先于时间戳被用户识别
- 长文本换行时，时间戳不得挤压角色名或遮挡正文
- 同一屏内消息之间的上下间距应统一

#### 9.2.5 验收口径

- 普通对话消息中，角色与时间戳处于同一行
- 普通消息块高度相较 v1 有明显收敛
- 在主流手机宽度下，消息流整体阅读密度提升，但不破坏聊天感
- 用户仍可清楚区分“你”和“小I”的发言

### 9.3 F8 投资团队工作进度反馈增强

#### 9.3.1 需求目标

让用户在长任务处理中持续感知“投资团队正在为自己工作”，而不是只看到抽象的系统处理中提示。

#### 9.3.2 功能规则

- 活跃 `task_status` 增加轻量转圈动效
- 已完成 `task_status` 增加完成标记
- 已用时间、预计剩余时间的提示更突出
- `task_status` 文案从系统状态升级为团队或成员工作进度反馈

#### 9.3.3 文案规则

`task_status` 须统一使用“谁在做什么”的表达，而不是技术状态表达。

建议结构：

- `<团队/成员> 正在 <动作>`
- `<团队/成员> 已完成 <动作>`
- `<团队/成员> 暂时中断 <动作>`

示例方向：

- `新闻分析师正在整理近期关键事件`
- `基本面分析师正在梳理盈利与估值`
- `风险分析师正在评估下行风险`
- `投资总监正在汇总结论`

文案约束：

- 优先使用业务可理解主体，不暴露底层技术身份
- 文案应专业、轻量、有工作感，不使用表演化表达
- 禁止出现 Agent 名、模型名、服务名、阶段 ID 等技术细节

#### 9.3.4 交互要求

- 活跃态显示轻量动态标记
- 完成态停止动效并显示完成标记
- 状态主文案优先表达主体和动作
- 已用时间和预计剩余时间的视觉层级低于主文案，但高于辅助说明
- 即使阶段没有切换，也应通过时间变化和轻量动态状态让用户感知任务仍在继续

#### 9.3.5 状态机要求

`task_status` 的状态定义需与 `tradingagents` 的 `stage` 和 `agent node` 对齐。

前端状态分为“阶段状态”和“节点状态”两层：

- 阶段状态：面向用户展示当前分析推进到哪一阶段，以 `stage_id` 为主键。
- 节点状态：面向用户补充说明当前阶段内“谁正在工作”，以 `node_id` 为辅助信息来源。

其中：

- `stage_id` 用于定义任务所处的大阶段，是前端主状态的唯一依据。
- `node_id` 用于描述当前阶段内具体正在运行的分析师、工具节点或管理节点，不单独作为任务阶段。

阶段状态至少支持以下状态：

| 状态 | 含义 | 展示要求 |
| --- | --- | --- |
| `pending` | 已发起，等待开始处理 | 展示等待中的状态文案，不展示完成标记 |
| `active` | 正在处理 | 展示动态标记、主体动作文案、已用时间和预计剩余时间 |
| `done` | 当前阶段完成 | 停止动效，展示完成标记和已完成文案 |
| `failed` | 任务失败或中断 | 停止动效，展示失败或中断文案，可提示稍后重试 |
| `stalled` | 长时间未更新但任务未明确失败 | 展示“仍在推进”或“处理时间较长”的兜底提示 |

#### 9.3.5.1 TradingAgents 阶段映射要求

前端需基于 `tradingagents` 输出的 `stage_id` 建立固定映射，不得自行推断新的业务阶段名称。

当前阶段映射如下：

| tradingagents stage_id | 前端阶段名称 | 对用户展示主体 | 对用户展示动作 |
| --- | --- | --- | --- |
| `analysts.market` | 市场分析 | 市场分析师 | 梳理价格走势与技术信号 |
| `analysts.social` | 情绪分析 | 情绪分析师 | 整理社交舆情与市场情绪 |
| `analysts.news` | 新闻分析 | 新闻分析师 | 整理近期关键事件与新闻影响 |
| `analysts.fundamentals` | 基本面分析 | 基本面分析师 | 梳理财务表现、盈利与估值 |
| `research.debate` | 研究讨论 | 研究团队 | 汇总多方观点并形成研究结论 |
| `trader.plan` | 交易计划 | 交易分析师 | 生成交易方案与执行思路 |
| `risk.debate` | 风险评估 | 风险团队 | 评估下行风险与仓位约束 |
| `portfolio.decision` | 投资决策 | 投资总监 | 输出最终投资决策 |

要求如下：

- 前端主文案以 `stage_id` 映射出的业务阶段和业务主体为准。
- 若后端后续扩展新的 `stage_id`，未映射的阶段前端应归入兜底阶段，不得直接暴露原始技术字段给用户。
- 对外展示时可使用更自然的业务文案，但不得改变阶段顺序和阶段语义。

#### 9.3.5.2 TradingAgents 节点映射要求

前端需识别 `node_id`，用于增强“谁正在工作”的感知，但不得让节点层级覆盖阶段层级。

当前节点映射原则如下：

| node_id 模式 | node_kind | 归属 stage_id | 前端展示用途 |
| --- | --- | --- | --- |
| `Market Analyst` | `agent` | `analysts.market` | 表示市场分析师正在工作 |
| `Social Analyst` | `agent` | `analysts.social` | 表示情绪分析师正在工作 |
| `News Analyst` | `agent` | `analysts.news` | 表示新闻分析师正在工作 |
| `Fundamentals Analyst` | `agent` | `analysts.fundamentals` | 表示基本面分析师正在工作 |
| `Bull Researcher` / `Bear Researcher` / `Research Manager` | `agent` | `research.debate` | 表示研究团队正在讨论与归纳观点 |
| `Trader` | `agent` | `trader.plan` | 表示交易分析师正在形成交易方案 |
| `Aggressive Analyst` / `Conservative Analyst` / `Neutral Analyst` | `agent` | `risk.debate` | 表示风险团队正在做多角度评估 |
| `Portfolio Manager` | `agent` | `portfolio.decision` | 表示投资总监正在输出最终结论 |
| `tools_*` | `tool` | 对应所属阶段 | 表示当前分析师正在调用数据或工具 |
| `Msg Clear *` | `utility` | 对应所属阶段 | 内部清理节点，前端默认不单独展示 |

要求如下：

- 当存在活跃节点时，前端可基于 `node_id` 生成更具体的“谁正在做什么”文案。
- `tool` 节点不单独展示技术名，应折叠为“正在收集数据”“正在补充信息”等业务表达。
- `utility` 节点不作为独立状态消息展示，避免暴露内部流程细节。
- 同一阶段内多个节点轮转时，前端保持一个阶段状态卡片，仅刷新文案，不追加多条碎片状态。

#### 9.3.6 状态更新规则

- 同一任务的 `task_status` 允许多次更新。
- 前端应以 `task_id + stage_id` 识别阶段状态更新，不得将同一任务的状态更新误展示为多个无关任务。
- 若后端提供 `stage.started` 事件，前端将对应阶段置为 `active`。
- 若后端提供 `stage.completed` 事件，前端将对应阶段置为 `done`，并切换到下一阶段。
- 若后端提供 `stage.failed` 事件，前端将当前阶段置为 `failed`。
- 若后端提供 `stage.stalled` 事件，前端将当前阶段置为 `stalled`，但仍保留“任务仍在推进”语义。
- 若后端未提供新阶段，但时间持续增加，前端仍需刷新已用时间展示。
- 当前任务完成后，最后一个阶段进入 `done`，不再展示进行中动效。
- 当前任务失败后，状态进入 `failed`，不再展示预计剩余时间。

#### 9.3.6.1 节点事件使用规则

若运行时提供 `node.started`、`node.completed`、`node.failed`、`node.stalled` 事件，前端按以下规则使用：

- `node.started`：更新当前阶段内的主体文案，例如“新闻分析师正在整理近期关键事件”。
- `node.completed`：不单独追加完成消息，仅用于内部推进判断。
- `node.failed`：若阶段未立即失败，可先保留阶段活跃态；若后续收到 `stage.failed`，再统一切为失败态。
- `node.stalled`：可作为当前阶段进入 `stalled` 的补充依据，但以 `stage.stalled` 为最终准。

#### 9.3.6.2 前端展示优先级

前端状态展示优先级如下：

1. 以 `stage` 事件决定当前任务处于哪个阶段。
2. 以 `stage` 状态决定当前阶段是 `pending`、`active`、`done`、`failed` 还是 `stalled`。
3. 以 `node` 事件补充当前阶段内的具体主体和动作文案。
4. 若 `node` 信息缺失，回退到 `stage_id` 对应的默认主体与动作文案。

#### 9.3.7 产品行为约束

- `task_status` 仍是状态反馈，不是聊天回复正文
- `task_status` 不承担结果卡片职责
- `task_status` 不展示技术监控式指标
- 不引入复杂百分比进度条
- 不展示底层多 Agent 编排图
- 不做技术监控面板式任务看板

## 10. 页面与交互说明

### 10.1 页面范围

本次需求仅影响当前会话页的消息流区，不新增页面。

### 10.2 消息流交互规则

- 用户发送消息后，消息流正常追加用户消息
- 若触发分析任务，消息流中出现对应 `task_status`
- 分析完成后，消息流承接 `小I` 的回复和结果卡片
- 若任务失败，消息流中展示 `小I` 错误消息和对应异常状态

### 10.3 历史消息回放规则

- 历史消息进入当前会话页时，按当前展示规则渲染
- 不要求迁移历史存储字段，但前端须根据消息类型完成统一展示
- 历史 `task_status` 若已完成，应显示为静态完成态

## 11. 边界与异常场景

### 11.1 网络异常

- 页面网络异常时，正在处理中的 `task_status` 不得无限保持正常进行中语义
- 若请求失败，可进入 `failed` 或 `stalled` 状态，并提示任务处理中断或等待恢复

### 11.2 页面离开与返回

- 用户离开页面再返回时，当前任务状态应按最新可获取状态恢复
- 若任务已完成，应显示完成态
- 若任务仍进行中，应恢复为活跃态并继续更新时间信息

### 11.3 多轮追问

- 用户在上一轮任务尚未结束时发起新的追问，需明确新旧任务状态归属
- 前端不得将不同任务的 `task_status` 混淆为同一条状态

### 11.4 长时间无更新

- 若任务长时间无新阶段回传，前端需提供“处理时间较长”类兜底提示
- 不得让页面长时间静止而使用户误以为系统挂起

### 11.5 错误消息展示

- `error` 类型消息应统一以 `小I` 口吻展示
- 错误消息应留在消息流中，便于用户理解上下文

## 12. 数据与接口要求

### 12.1 前端渲染最低数据要求

前端渲染本需求至少需要以下字段能力：

| 字段 | 说明 | 是否必需 |
| --- | --- | --- |
| `message_type` | 消息类型，用于映射角色与模板 | 是 |
| `message_id` | 消息唯一标识 | 是 |
| `created_at` | 消息时间戳 | 是 |
| `task_id` | 任务唯一标识，用于关联状态更新 | `task_status` 必需 |
| `task_state` | 任务状态值 | `task_status` 必需 |
| `status_text` | 状态展示文案 | `task_status` 推荐 |
| `elapsed_time` | 已用时间 | `task_status` 推荐 |
| `eta` | 预计剩余时间 | `task_status` 可选 |

### 12.2 前后端协作要求

- 前端基于 `message_type` 决定展示角色与模板
- 后端如已有 role 字段，可保留；本期以前端映射规则为准
- 后端如提供任务状态流，需保证 `task_id` 稳定
- 若后端暂不提供 `eta`，前端可仅展示已用时间，不阻塞本期上线

## 13. 设计与开发要求

### 13.1 设计要求

- 明确 `你`、`小I`、`task_status` 三类展示形态差异
- 给出普通消息、结果卡片、错误消息、任务状态的组件规范
- 给出活跃态、完成态、失败态、长时间处理中态的视觉方案
- 给出不同屏宽下的适配规则

### 13.2 前端开发要求

- 建立消息类型到展示角色、模板的稳定映射
- 保证消息列表在历史回放与实时更新场景下展示一致
- 处理 `task_status` 的更新、替换、完成、失败逻辑
- 保证时间信息刷新不造成明显页面抖动

### 13.3 测试要求

- 覆盖消息类型渲染正确性
- 覆盖任务状态流转正确性
- 覆盖网络异常、页面返回、多任务并存、历史回放等边界情况
- 覆盖主流手机尺寸下的视觉表现

## 14. 验收标准

### 14.1 功能验收

- 用户在会话页能稳定感知系统发言角色为 `小I`
- `text`、`ticker_resolution`、`insight_reply`、`summary_card`、`error` 五类消息在前端展示层角色统一
- `task_status` 保持独立状态反馈定位，不与 `小I` 混淆
- 普通对话消息中，角色与时间戳位于同一行
- 活跃态 `task_status` 具有明确主体与动作表达
- 完成态 `task_status` 有完成标记，且不再保留进行中动效
- 异常态 `task_status` 不继续展示进行中语义

### 14.2 体验验收

- 消息流整体阅读密度较 v1 提升，移动端阅读负担降低
- 用户可清楚区分“你”和“小I”的发言
- 用户在长任务处理中不会因页面静止而误以为系统挂起
- 全部状态文案不暴露内部技术实现细节
- 整体体验比 v1 更像一个有统一人格、有持续反馈的投研对话工作台

## 15. 风险与注意事项

### 15.1 角色边界混淆风险

若不明确区分 `小I` 与 `task_status` 的职责，容易让用户误以为所有状态消息都是聊天回答，从而削弱“团队正在工作”的感知。

### 15.2 过度人格化风险

如果 `task_status` 文案过度拟人化或戏剧化，可能削弱专业投研产品的可信度，因此文案应保持专业、轻量、有工作感。

### 15.3 过度技术化风险

若将内部技术阶段直接映射到用户界面，会破坏产品的专业感与可理解性。本需求须优先让用户理解“谁在为自己做什么”，而不是系统内部如何调度。

### 15.4 状态一致性风险

若 `task_id`、`task_state` 或状态更新时序不稳定，前端容易出现状态错乱、重复展示或完成态未落定的问题。

## 16. 上线后的观察指标

建议上线后至少观察以下指标，用于验证体验升级效果：

- 长任务期间的会话离开率
- 分析进行中用户重复提交或重复点击比例
- 首次结果返回前的平均停留时长
- 长任务期间错误退出或中断比例

## 17. 版本结论

v1.1 为 v1 的增量升级版本，不改变产品主路径，仅围绕以下三点完成体验增强：

- 消息角色统一
- 消息元信息紧凑化
- 任务状态表达升级

上线后应达到以下整体结果：

- 用户能够明确分辨“你”和“小I”
- 系统类对话输出具备统一前台角色体验
- 消息流更紧凑，移动端阅读负担更低
- 长分析过程中，用户能明确感知投资团队正在持续推进工作
