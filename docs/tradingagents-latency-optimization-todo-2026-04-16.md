# TradingAgents 分析耗时优化 TODO（2026-04-16）

## 1. 背景

当前用户发起一次 `TradingAgents` 分析后，需要等待超过 10 分钟才能看到完整分析报告，等待时间偏长，已经影响移动端和对话式产品体验。

基于本机日志对单次真实 run 的拆解，当前耗时的主要结论如下：

- 样本 run：`SNDK-b9793ec570b6`
- 总耗时：约 `645.5s`，即 `10分46秒`
- 工具调用总耗时约 `14.4s`
- 主要耗时集中在多轮串行 LLM 推理，而不是数据接口或 IO

按阶段拆解，大头主要在：

- `research.debate`：约 `167.7s`
- `analysts.market`：约 `107.4s`
- `risk.debate`：约 `91.6s`
- `analysts.news`：约 `77.4s`
- `analysts.fundamentals`：约 `76.3s`
- `analysts.social`：约 `70.9s`

按 agent 拆解，大头主要在：

- `Market Analyst`：约 `105.9s`
- `News Analyst`：约 `71.7s`
- `Fundamentals Analyst`：约 `71.3s`
- `Social Analyst`：约 `68.7s`
- `Bear Researcher`：约 `64.0s`
- `Bull Researcher`：约 `53.5s`
- `Research Manager`：约 `50.1s`
- `Portfolio Manager`：约 `42.8s`

结论很明确：

- 当前核心瓶颈不是工具层，而是 agent 之间的多轮串行 LLM 调用
- 如果只优化工具层，收益会比较有限
- 真正需要优先优化的是编排方式、输出策略、上下文体积和模型分层

---

## 2. 目标

建议先把目标拆成两层：

- 用户首次看到“有价值内容”的时间显著下降
- 完整报告总耗时逐步下降

建议目标区间：

- 首屏可见结果：`1-3 分钟`
- 标准完整分析：`3-6 分钟`
- 深度模式完整分析：`6-10 分钟`

## 3. 产品 TODO

以下条目建议你逐条回复，可以直接在每条后面补：

- `结论：接受 / 拒绝 / 待定`
- `备注：...`

### P1. 引入“快报优先，完整报告后补”的输出策略

- TODO：将当前“一次性等完整报告产出后再展示”改为“先给快报，再持续补全”
- 目标：用户在 30-60 秒内先看到结论、评级、关键信号和主要风险
- 说明：完整报告、辩论明细、章节化内容可以异步补齐
- 价值：显著改善等待体感，即使总时长不立刻减半，用户也不会长时间无反馈
- 需要决策：
  - 快报是否必须包含明确评级
  - 快报是否允许后续结论修正

### P2. 提供分析深度分档

- TODO：提供至少三档分析模式，例如 `快` / `标准` / `深度`
- 目标：把时延和分析深度显式产品化，减少用户预期错配
- 建议：
  - `快`：2-3 分钟，适合先看方向
  - `标准`：4-6 分钟，适合作为默认模式
  - `深度`：8-12 分钟，适合高投入场景
- 需要决策：
  - 默认档位是否用 `标准`
  - 哪些入口允许用户切换档位

### P3. 优化任务进行中的体验

- TODO：在等待过程中展示“当前阶段 + 已完成模块 + 当前发现摘要 + 预估剩余时间”
- 目标：让等待期变成“可感知的推进过程”，而不是单纯 loading
- 说明：用户更容易接受“正在生成高价值内容”，而不是“系统卡住了”
- 需要决策：
  - 是否展示 agent 名称
  - 是否展示当前发现的中间结论

### P4. 支持“先结论，后追问”的交互路径

- TODO：允许系统优先返回结论卡，详细报告按需展开或继续追问
- 目标：降低首次回答长度和首次等待时间
- 说明：很多用户并不需要一上来就读完整报告
- 需要决策：
  - 首答是否默认只给摘要版
  - 报告详情是按钮展开还是自然继续对话

### P5. 支持近期分析结果复用

- TODO：针对短时间内重复请求同一 ticker 的场景，支持“直接查看上次结果 + 增量更新”
- 目标：减少重复计算
- 说明：对热门 ticker 的重复分析可能很多
- 需要决策：
  - 复用窗口设为多久
  - 是默认复用，还是提示用户确认

## 4. 技术 TODO

### T1. 将 analyst 阶段从串行改为并行

- TODO：让 `market / social / news / fundamentals` 四个 analyst 并发执行，而不是串行
- 现状问题：当前四个 stage 基本是相加关系，直接把总耗时堆高
- 预期收益：这是最可能带来“分钟级下降”的优化项之一
- 风险：
  - 汇总层需要等待所有 analyst 结果
  - 共享上下文和共享缓存要设计清楚
- 建议优先级：最高

### T2. 将 analyst 的执行方式从多轮 ReAct 改为“先取数，后总结”

- TODO：减少 `agent -> tool -> agent -> tool -> ...` 的往返轮数
- 建议方向：
  - 第一轮先规划所需数据
  - 批量取数
  - 最后一次性生成总结
- 现状问题：单个 analyst 有明显的多轮 LLM 调用，且上下文越滚越大
- 预期收益：可以显著压缩单个 analyst 的执行时间
- 建议优先级：最高

### T3. 限制每个 analyst 的最大工具回合数

- TODO：为每类 analyst 设置合理的最大工具回合数和提前终止条件
- 现状问题：部分 agent 在没有明显新信息的情况下继续多轮调用
- 建议方向：
  - 设置最大工具轮数
  - 若新增信息不足，提前退出
  - 若工具结果为空或重复，直接进入总结
- 建议优先级：高

### T4. 压缩 prompt 和历史上下文体积

- TODO：避免把完整历史、完整工具原文不断回灌给后续轮次
- 现状问题：日志中可见同一 agent 的输入字符数逐步膨胀，这会直接拖慢后续推理
- 建议方向：
  - 工具结果先结构化摘要
  - 辩论阶段只携带关键证据摘要
  - 避免把整份章节报告反复带入后续节点
- 建议优先级：高

### T5. 将 research debate 改为单轮对照或并行生成

- TODO：重新设计 `Bull Researcher / Bear Researcher / Research Manager` 的流程
- 现状问题：当前研究辩论是明确串行链路，耗时约 `167.7s`
- 建议方向：
  - Bull / Bear 并行生成观点
  - Research Manager 汇总裁决
  - 或者合并为一个 agent 输出“多空观点 + 裁决”
- 需要决策：
  - 是否保留“多 agent 辩论感”
  - 是否允许牺牲部分仪式感来换时延
- 建议优先级：高

### T6. 将 risk debate 改为更短路径

- TODO：重新设计 `Aggressive / Conservative / Neutral / Portfolio Manager` 的流程
- 现状问题：当前 risk 阶段也是串行，实际耗时约 `91.6s`，portfolio 决策约 `42.8s`
- 建议方向：
  - 三个 risk analyst 并行产出
  - Portfolio Manager 汇总
  - 或压缩为两个风险视角
- 建议优先级：高

### T7. 做模型分层

- TODO：不要让所有节点都使用同一档模型能力
- 建议方向：
  - analyst 初稿和风险子角色用快模型
  - `Research Manager` / `Portfolio Manager` 用更强模型
- 现状问题：当前所有主要节点都在消耗较长 LLM 推理时间
- 预期收益：在不大改结构的前提下，也能拿到一部分时延收益
- 建议优先级：高

### T8. 将最终报告生成与最终结论拆开

- TODO：把“最终决策”与“完整报告排版/落盘”解耦
- 目标：先返回 `final_trade_decision + executive_summary`，报告文件稍后再补
- 价值：显著缩短用户第一次得到答案的时间
- 建议优先级：高

### T9. 建立 run 内数据缓存与跨请求热点缓存

- TODO：对同一 ticker 的基础行情、新闻、基本面结果做复用
- 现状问题：工具耗时占比不高，但复用后可以进一步减少重复取数
- 建议方向：
  - 单次 run 内共享缓存
  - 短时间窗口内跨 run 热点缓存
- 建议优先级：中

### T10. 对常见 ticker 支持增量更新而不是全量重跑

- TODO：建立“已有分析结果 + 最新市场增量”的更新机制
- 目标：减少重复执行完整 analyst 链路
- 适用场景：热门 ticker、用户高频追问、短时间内重复发起分析
- 建议优先级：中

### T11. 加强性能观测指标

- TODO：把耗时分析从手工查日志变成常规观测
- 建议补充指标：
  - 单 run 总耗时
  - 首条可见结果耗时
  - 各 stage 耗时
  - 各 agent 耗时
  - 单 agent LLM 次数
  - prompt 字符数增长趋势
  - 工具调用次数和命中缓存比例
- 目标：后续优化可以量化，不靠体感
- 建议优先级：高

### T12. 引入性能基线和回归守护

- TODO：对关键路径建立性能基线，避免后续越改越慢
- 建议方向：
  - 固定样本 ticker 的离线 benchmark
  - CI 或定期任务做性能趋势比对
  - 若 stage 或 agent 时长异常增长则告警
- 建议优先级：中

## 5. 建议实施顺序

建议先按“低争议、高收益”的顺序推进：

### 第一批：先改善用户感知

- P1. 快报优先，完整报告后补
