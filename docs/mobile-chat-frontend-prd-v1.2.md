# 对话式移动端前端需求说明书 v1.2

## 1. 文档信息

- 文档名称：对话式移动端前端需求说明书 v1.2
- 版本：v1.2
- 当前方案：Insight Agent 对话效率提升方案
- 文档类型：需求说明书
- 面向对象：产品、前端、后端、设计、测试
- 适用范围：手机浏览器 H5 当前会话页在 v1.1 基础上的追问体验升级版本

## 2. 背景与问题定义

### 2.1 当前版本基础能力

截至 v1.1，移动端 H5 已具备以下基础能力：

- 用户可在同一会话页发起分析、查看任务进度、阅读分析结果与继续追问
- 系统消息、结果卡片与错误提示已统一到更稳定的前台人格与消息流结构
- 分析过程中的 `task_status` 已能体现投资团队工作推进状态

### 2.2 当前问题

在 v1.1 之后，用户继续追问阶段暴露出新的体验瓶颈：

1. `insight_reply` 的等待时间仍然偏长，用户在追问后难以快速获得回应。
2. `insight_reply` 的内容偏长、偏完整解释，阅读成本较高。
3. 当前回复更像“分析报告的二次解读”，而不像围绕用户当前问题展开的自然对话。
4. 用户在连续追问时，容易从“聊天”体验退化为“不断阅读长篇文本”的体验。

### 2.3 根因判断

当前 `insight_reply` 背后的 Insight Agent 仍然偏向“章节探索型解读 Agent”：

- 优先考虑读取报告章节，再组织答案
- 默认输出较完整的解释型内容
- 对简单追问与复杂追问缺少一致的对话效率策略
- 尚未显式建立“用户当前问题 ↔ 分析后生成的多份 report”之间的回答关联策略

其结果是：

- 简单问题也可能等待较久
- 回答内容天然倾向于过长
- 用户感知到的是“又收到一段报告说明”，而不是“被直接回答”
- 当问题天然对应某一类分析材料时，系统仍可能按“大报告”思路处理，而不是优先命中最相关的那份 report

### 2.4 本次改版定位

v1.2 不改变当前会话页的主结构，也不取消现有深度解读能力，而是聚焦升级 `insight_reply` 对应的 Insight Agent，使其从“研究型回答器”演进为“更高对话效率的解读助手”。

## 3. 产品定位

v1.2 延续 v1 与 v1.1 的产品定位：

**一个面向手机浏览器的轻量投研对话工作台。**

本次升级重点强化以下感知：

1. 用户提问后，系统应优先直接回应当前问题。
2. 系统在需要时仍可展开深入解读，但不默认一次性讲完。
3. 连续追问应更像对话，而不是反复触发长篇报告式输出。

## 4. 目标与非目标

### 4.1 产品目标

- 提升 `insight_reply` 的对话效率，缩短用户追问后的有效反馈等待感
- 让 Insight Agent 优先回答当前问题，而不是优先展开完整解释
- 降低单次 `insight_reply` 的默认阅读负担，提升连续追问意愿
- 保留深度解读能力，但将其变为按需展开而非默认展开

### 4.2 非目标

- 不重做登录、抽屉、当前会话页的页面结构
- 不替换 v1.1 已完成的 `task_status`、`小I`、消息流紧凑化能力
- 不取消现有报告解读能力
- 不要求在本期引入新的独立报告页或问答页
- 不要求本期重做 TradingAgents 的主分析图结构
- 不要求本期整体缩短主分析任务从发起到 `summary_card` 生成的完整耗时
- 不要求本期对 TradingAgents 各 analyst、debate、risk、portfolio 阶段进行流程裁剪或重排
- 不要求本期引入新的缓存体系、预计算体系或流式输出体系
- 不要求本期为所有追问场景设计多套独立问答产品形态
- 不以“压缩字数”作为唯一目标，也不以“所有回复都极短”作为目标
- 不以牺牲答案依据、上下文一致性和解读可信度来换取表面上的回复更短或更快

## 5. 适用范围与影响范围

### 5.1 适用范围

本需求适用于移动端 H5 当前会话页中的以下部分：

- `insight_reply` 消息生成逻辑
- `insight_reply` 默认展示内容
- 用户在 `report_ready` 与 `report_explaining` 状态下的继续追问体验

### 5.2 受影响模块

- `ta_service/services/conversation_service.py`
- `ta_service/services/report_insight_agent.py`
- `ta_service/services/report_context_loader.py`
- `mobile_h5` 当前会话页对 `insight_reply` 的展示策略

### 5.3 不受影响范围

以下能力默认保持 v1.1 不变：

- 标的识别与确认流程
- 主分析任务的发起方式
- `task_status` 状态反馈体系
- 分析完成后 `summary_card` 的主承载角色

## 6. 角色与术语定义

### 6.1 角色定义

- `你`：用户在会话中的展示角色
- `小I`：前台统一对话角色
- `Insight Agent`：用于回答分析完成后用户追问的系统解读 Agent

### 6.2 术语说明

- `insight_reply`：针对分析报告内容的继续追问回复消息
- 对话效率：系统优先直接回答当前问题、减少不必要展开、降低等待与阅读成本的能力
- 回答优先：系统优先尝试回答问题，仅在必要时补充读取更多上下文
- 按需检索：系统仅在现有上下文不足以支撑回答时，才读取更详细的报告章节
- 深度展开：围绕某个问题提供更完整的依据、推理和细节说明
- 多 report：指一次分析完成后按不同分析角色或环节生成的多份报告材料，如 market、news、fundamentals、sentiment、research、risk、decision 等
- report 关联：系统先判断用户问题更应由哪一份或哪几份 report 支撑回答，再决定默认取材范围与补充读取路径

## 7. 需求原则

### 7.1 回答优先

Insight Agent 应优先回应用户当前问题，而不是默认组织完整解读材料。

### 7.2 按需展开

系统应保留深入解释能力，但默认不一次性输出全部说明。

### 7.3 保持对话感

用户连续追问时，体验应接近对话往返，而不是阅读多轮长篇结果。

### 7.4 不牺牲可信度

提升对话效率不能以牺牲答案依据和上下文一致性为代价。

## 9. 详细功能需求

### 9.1 F9 Insight Agent 对话效率提升

#### 9.1.1 需求目标

将当前偏研究型、偏长文本、等待时间较长的报告解读能力，升级为一个回答优先、按需检索、可持续追问的高对话效率 Insight Agent。

#### 9.1.2 实现路径定义

本功能应明确基于 LLM Agent 的实现路径落地，而不是基于固定规则表或纯模板拼接实现。

本期的 `Insight Agent` 应被定义为一个“带最小工具能力的报告解读 Agent”，其核心链路为：

1. 接收当前追问与会话上下文
2. 判断现有上下文是否足以回答
3. 不足时按需调用 report 读取能力补充材料
4. 基于已知材料生成最终回答

该路径要求系统保留 Agent 的判断空间，不将“问题类型 -> report 集合”固化为硬编码产品规则。

#### 9.1.3 Agent 输入上下文要求

`Insight Agent` 的输入上下文应至少包含以下信息：

- 当前会话 ID
- 用户当前问题
- 当前分析标的与分析日期
- 最近若干轮用户与助手对话历史
- 当前分析结果可用的 report 章节列表
- 可选的 report 根目录或章节读取入口
- 当磁盘 report 不可用时的降级摘要材料

上下文组织原则如下：

- 优先提供轻量、结构化、可判断的上下文，而不是一次性拼接全部报告正文
- 应让 Agent 知道“有哪些 report 可用”，而不是默认把所有报告文本直接塞入 prompt
- 应保留多 report 的边界信息，便于 Agent 自主判断取材范围
- 当完整 report 不可用时，允许使用 `summary_card` 或执行摘要作为降级上下文

#### 9.1.4 Agent 工作流要求

`Insight Agent` 的默认工作流应定义为：

1. 基于当前问题、最近对话历史与可用章节目录，先尝试理解问题意图
2. 判断该问题是否可由现有轻量上下文直接回答
3. 若可直接回答，则直接生成回答，不额外读取章节
4. 若不可直接回答，则按需读取最相关的一个或少数几个 report 章节
5. 若仍不足，再继续补充最小必要章节
6. 在章节探索结束后，进入单独的最终回答阶段
7. 最终回答阶段禁止继续读章节，只能基于已获取材料作答

该工作流的目标不是减少 Agent 判断，而是限制无必要的探索轮次与无边界展开。

#### 9.1.5 章节读取与工具调用要求

本期允许 `Insight Agent` 使用“读取 report 章节”的工具能力，但该能力应满足以下约束：

- 工具能力仅用于按需读取章节，不承担独立回答职责
- Agent 应先决定是否需要读章节，再决定读哪个章节
- Agent 不应默认先读取多个章节后再思考
- 单次追问中的工具调用轮次应有明确预算控制
- 单轮中可调用的章节数应有上限，避免无边界遍历
- 如果工具预算耗尽，Agent 应基于已读材料收敛出最终回答

工具调用的产品意图是“补充回答依据”，而不是“替代回答判断”。

#### 9.1.6 多 report 使用原则

本期应明确支持“分析后生成多份 report”的现实结构，但不将其设计为静态命中表。

多 report 的使用原则如下：

- Agent 应将多份 report 视为可选依据池，而不是默认全部加载的整体报告
- 当问题明显聚焦某一分析维度时，应优先使用最相关的单份 report 或少数几份 report
- 当问题要求比较、归因、冲突解释、结论来源说明或综合判断时，可联动多份 report
- 当多个 report 之间存在一致、互补或冲突关系时，Agent 可在回答中表达这种关系
- 不要求产品层为所有问题维护固定的 report 映射表

这一定义的重点是“让 Agent 基于问题做判断”，而不是“让系统预先替 Agent 做静态路由”。

#### 9.1.7 最终回答生成要求

在完成必要的上下文补充后，系统应进入独立的最终回答生成阶段。

该阶段要求如下：

- 不再允许继续调用章节读取工具
- 仅基于当前问题、最近对话历史与已读取材料回答
- 优先直接回应用户当前问题
- 回答结构应优先为“直接回答 + 少量关键点”
- 当已有材料不足时，应明确说明“根据本次分析报告，无法回答该问题”
- 不得引入报告之外的信息，不得凭空补充外部知识

这一步的目标是把 Agent 从“探索模式”切回“回答模式”。

#### 9.1.8 输出体验要求

- `insight_reply` 应更像对话回应，而不是报告补丁
- 用户应先感知到“问题被回答了”
- 默认回答应适合移动端阅读，不应天然长文化
- 默认回答应保留继续追问空间，而不是一次性展开全部依据
- 当用户明确追问原因、依据、差异、展开说明时，后续轮次可逐步增加深度
- 连续追问的节奏应保持“问题驱动”，而不是“报告驱动”

#### 9.1.9 可观测性与降级要求

- 系统应能够记录本轮回答是否可回答、读取了哪些章节、是否触发了工具预算上限
- 当 report 不可用但存在摘要材料时，应走降级路径回答
- 当 LLM 或章节读取能力不可用时，应返回明确、可理解的失败提示，而不是空回复
- 当最终回答为空或材料不足时，应回退到稳定兜底回复

#### 9.1.10 非功能要求

- 本期不要求将所有追问都压缩为极短回复
- 本期不要求 Insight Agent 放弃深度解释能力
- 本期不要求完全取消章节读取机制
- 本期不要求通过静态规则表替代 Agent 判断
- 本期重点是提升“回答效率”和“对话节奏”，而非单纯压缩字数

## 10. 页面与交互影响

### 10.1 页面范围

本次需求不新增页面，仍然在当前会话页完成。

### 10.2 消息流交互规则

- 用户在 `report_ready` 或 `report_explaining` 状态下继续追问
- 系统返回 `insight_reply`
- `insight_reply` 默认应更轻量、更直接
- 如用户继续深入提问，后续回复可逐步展开

## 11. 后端能力要求

### 11.1 上下文构建能力要求

后端需先完成 `Insight Agent` 所需上下文的构建，再进入 LLM 调用阶段。

上下文构建阶段至少应提供以下能力：

- 从当前会话中提取最近若干轮 user / assistant 对话历史
- 从当前任务中提取分析标的、分析日期、任务 trace 信息
- 基于 trace 信息识别当前可用的 report 章节列表
- 在完整 report 可用时，仅向 Agent 暴露章节目录与读取入口
- 在完整 report 不可用时，提供基于 `summary_card` 或执行摘要的降级上下文

该阶段的职责是“构建可判断的输入”，而不是“提前替 Agent 完成回答决策”。

### 11.2 Report 上下文装载能力要求

后端需提供独立的 report 上下文装载能力，用于承接多 report 的存储与按需读取。

该能力至少应支持：

- 列出当前分析结果中实际存在的 report 章节
- 按章节名读取单个 report 内容
- 保留各 report 的独立边界，不将所有报告默认拼接成单段大文本
- 对上下文读取量设置上限，避免单轮追问加载过多内容
- 在 report 文件不存在、内容为空或读取失败时返回稳定结果

该能力的目标是“支持 Agent 按需取材”，而不是“预先把所有材料一次性塞给 Agent”。

### 11.3 Agent 执行能力要求

`ReportInsightAgent` 需按 LLM Agent 方式执行，而不是按固定规则表或模板路由执行。

执行阶段至少应具备以下能力：

- 基于当前问题、对话历史与可用章节目录理解用户意图
- 判断现有轻量上下文是否足以回答
- 在必要时发起章节读取工具调用
- 在单 report 与多 report 之间自主决定取材范围
- 在已有材料足够时直接回答，不继续探索
- 在材料不足时返回明确的“根据本次分析报告，无法回答该问题”类结果

该阶段允许 Agent 自主判断问题与 report 的关联，但不要求产品层维护静态问题映射表。

### 11.4 工具调用控制要求

后端需对 Agent 的章节探索行为提供明确的预算控制。

控制要求如下：

- 工具调用应有轮次上限
- 每轮可调用的章节数应有上限
- Agent 超出预算后不得继续无限探索
- 当预算耗尽时，应进入最终回答阶段而不是继续尝试读取
- 工具调用失败时，不得导致整轮回答进入不可恢复状态

该机制的目标是限制无边界探索，同时保留必要的 Agent 灵活性。

### 11.5 最终回答阶段要求

后端需将“章节探索阶段”和“最终回答阶段”明确拆开。

最终回答阶段要求如下：

- 禁止继续调用章节读取工具
- 仅允许基于当前问题、最近对话历史与已读取章节材料生成答复
- 回答应优先直接回应问题，而不是继续组织长篇背景
- 若材料不足，应明确给出不可回答结论
- 最终回答应输出为稳定的 `insight_reply` 文本结果

该阶段的产品意义是让 Agent 在探索结束后收敛为稳定答复，而不是继续停留在搜索状态。

### 11.6 降级与容错要求

后端需提供明确的降级路径，保证在 report、工具或 LLM 不完整可用时，系统仍有稳定行为。

至少包括：

- 无 traceDir 且无完整 report 时，允许退化为基于 `summary_card` 的回答路径
- 无任何可用报告材料时，返回“当前会话暂无可用分析报告内容”的明确提示
- LLM 不可用时，返回统一、可理解的服务不可用提示
- 最终回答为空时，回退到稳定兜底回复
- report 读取失败时，允许在部分材料可用的情况下继续回答

### 11.7 可观测性要求

后端需为该 Agent 链路提供最小必要的调试与审计信息。

至少应记录：

- 当前问题是否被判定为可回答
- 本轮回答读取了哪些 report 章节
- 使用了多少轮工具调用
- 是否触发了工具预算上限
- 是否走了降级路径

这些信息主要用于调试、质量观察与后续策略优化，不要求直接暴露给前端用户。

## 12. 前端能力要求

### 12.1 展示要求

- `insight_reply` 的默认展示应服务于快速阅读
- 前端不应将 `insight_reply` 强制塑造成长篇正文阅读模式
- 若后端后续提供更结构化的回答结果，前端应具备承接能力

具体要求如下：

- `insight_reply` 在默认形态上应仍被渲染为聊天消息，而不是结果卡片或报告正文块
- 前端应优先突出“当前问题的直接回应”，而不是通过过强的标题、分段、装饰把其渲染成新报告
- 当回答较短时，应保持消息形态自然，不为“看起来像卡片”而增加额外容器复杂度
- 当回答较长时，应优先通过排版控制可读性，而不是默认切换到报告阅读模式
- `summary_card` 与 `insight_reply` 的视觉角色应保持分工：
  - `summary_card` 继续承担分析完成后的主结果承载
  - `insight_reply` 继续承担追问阶段的对话式解释承载

### 12.2 体验要求

- 用户连续追问时，应感知消息流仍保持聊天节奏
- `insight_reply` 不应明显破坏当前会话页的对话式阅读体验

具体要求如下：

- 用户发起追问后，新增的 `insight_reply` 应与上下文消息自然衔接，不应在视觉上突然切换产品模式
- 连续多轮追问时，消息流应保持“提问 - 回答 - 继续提问”的对话节奏
- 前端不应通过过重的分隔、折叠或章节化样式放大“报告感”
- 用户应能快速识别这一轮回答已经回应了当前问题，而不是又开始阅读新的长篇说明
- 若后续轮次变得更深入，前端也应保持相同消息范式，不因内容变长就切换为另一种页面结构

### 12.3 结构兼容要求

前端应兼容当前文本型 `insight_reply`，同时为后续更结构化的回答结果保留承接空间。

兼容要求如下：

- 当前版本下，前端需稳定承接纯文本或 markdown 型 `insight_reply`
- 若后端后续补充回答来源、章节引用、回答结构等字段，前端应可在不重构消息流骨架的前提下扩展展示
- 即使后端未来返回“直接回答 + 要点 + 来源章节”这类结构，前端也不应将其渲染成新的报告页面
- 结构化字段若存在，应服务于“增强回答可读性与可信度”，而不是引入新的阅读负担

### 12.4 交互节奏要求

前端承接 Agent 回答时，应优先保护追问链路的连续性。

要求如下：

- 用户在 `report_ready` 与 `report_explaining` 状态下继续追问时，输入、发送、消息追加的反馈应保持轻量直接
- 新的 `insight_reply` 到达后，应按普通聊天消息插入当前消息流
- 前端不应因 `insight_reply` 的来源是 Agent，就额外插入技术性过渡消息
- 当回答不可用或服务降级时，前端应将其作为自然语言回复承接，而不是暴露后端内部原因

### 12.5 前端不应承担的职责

本期前端不应承担以下职责：

- 不在前端用静态规则表判断“问题属于哪类 report”
- 不在前端重建 Agent 的多 report 选择逻辑
- 不在前端推断回答是否可回答
- 不在前端将多个 report 拼接后自行生成总结
- 不在前端新增独立的问答页、报告页或章节页来承接本期追问能力

## 13. 验收标准

### 13.1 功能验收

- `insight_reply` 在多数简单追问场景下，应能先给出直接回答
- `insight_reply` 不应在所有追问场景下都输出长段解释文本
- 对于复杂追问，Insight Agent 仍应能够提供更深入解读
- Insight Agent 的回答不应脱离本次分析报告上下文
- Insight Agent 在必要时才读取更详细章节内容

### 13.2 体验验收

- 用户追问后，默认感受到的是“被回应”，而不是“收到一篇新报告”
- 连续追问时，阅读负担较当前版本下降
- 回复节奏更接近对话，而不是每轮都进入长文阅读
- 用户在需要深入信息时，仍能通过继续追问获得更完整说明

## 14. 风险与注意事项

- 若只压缩回复长度而不调整 Agent 工作策略，可能导致“短但空”的体验
- 若过度压制解释深度，可能损害答案可信度
- 若仍保留当前偏重的章节探索机制，可能无法明显改善等待感
- 本期应重点避免“形式上更短，但本质上仍是报告式回应”的伪优化

## 15. 版本结论

v1.2 的重点不是单纯让 `insight_reply` 变短，而是让 Insight Agent 成为一个更高对话效率的解读助手。

其核心定义为：

**回答优先、按需检索、逐步展开、保持可信。**
