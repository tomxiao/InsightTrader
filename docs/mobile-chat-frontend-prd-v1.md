# 对话式移动端前端一期 PRD v1

## 1. 文档信息

- 文档名称：对话式移动端前端一期 PRD
- 版本：v1
- 当前方案：统一对话页方案
- 面向对象：产品、前端、后端、设计、测试
- 适用范围：手机浏览器 H5 前端一期

## 2. 背景

当前项目已有一套面向桌面管理台的前端，适合配置管理、后台操作和复杂数据查看，但不适合手机浏览器上的高频使用场景。

移动端一期不追求复制桌面端全部能力，而是聚焦用户最核心的两类任务：

1. 个股分析发起
2. 分析结果解读与继续追问

为降低移动端 UI/UX 复杂度，一期不再采用“首页、线程页、历史页、我的页”并列的传统结构，而是统一为：

**登录后直接进入当前会话页，在一个会话中完成分析发起与结果解读；新建会话、历史切换和账户操作统一收纳到左侧抽屉。**

## 3. 产品定位

对话式移动端前端一期定位为：

**一个面向手机浏览器的轻量投研对话工作台。**

它不是桌面后台的缩小版，而是一个更适合移动场景、更聚焦单任务闭环的使用端产品。

### 3.1 目标用户

- 普通登录用户
- 需要在手机上快速发起分析、查看结论、继续追问的用户
- 已在桌面端完成复杂配置，移动端主要用于“使用”和“消费结果”

### 3.2 一期价值主张

- 用自然语言替代表单式操作
- 用统一会话承载分析过程与结果解释
- 用结构化卡片降低移动端阅读负担
- 用左侧抽屉承载导航，减少主界面干扰

## 4. 目标与非目标

### 4.1 一期目标

- 支持用户在手机浏览器中完成登录、发起分析、查看结果、继续追问
- 建立“登录 -> 当前会话 -> 历史回看 -> 再次追问”的最小闭环
- 降低页面数量与导航复杂度
- 尽量复用现有后端 API、鉴权体系与报告能力

### 4.2 一期非目标

- 不复制桌面端后台管理能力
- 不实现复杂配置管理
- 不实现完整模拟交易终端
- 不实现图表密集型工作台与大表格后台页
- 不将产品做成开放域通用聊天助手
- 不以管理员或运维场景为核心

## 5. 核心设计原则

### 5.1 对话优先

用户优先通过自然语言发起任务，而不是填写复杂表单。

### 5.2 单会话承载主任务

“个股分析”和“报告解读”不是两个页面，而是同一会话中的两个阶段。

### 5.3 少页面、少层级

一期尽量将主体验收敛到一个核心页面，其他能力通过抽屉和轻量菜单承载。

### 5.4 结果卡片化

系统输出不能只有纯文本，必须提供结构化卡片承载关键结论。

### 5.5 默认值优先

能自动识别的参数不要求用户手动选择，必要时仅做轻量追问。

### 5.6 适合中断恢复

手机端用户经常切后台、断网、刷新页面，因此要优先支持任务恢复和历史回看。

## 6. 核心方案概述

### 6.1 登录后默认落点

用户登录成功后，默认直接进入：

- `当前会话页`

当前会话页既承担原本“首页”的欢迎和发起作用，也承担原本“线程页”的处理和解读作用。

### 6.2 左侧抽屉承担的职责

左侧抽屉用于承载辅助导航和会话管理，包括：

- `+ 开启新对话`
- `历史会话列表`
- `底部账户入口`

左侧抽屉不是一个独立业务页面，而是当前会话页的导航容器。

### 6.3 “我的”简化方案

一期不单独做完整“我的页”，而是将其简化为抽屉底部的账户菜单。

账户菜单采用轻量形态：

- 抽屉底部展示头像、昵称、更多按钮
- 点击后弹出轻量菜单

一期建议菜单项仅保留：

- `系统设置`
- `帮助与反馈`
- `退出登录`

## 7. 一期功能范围

### 7.1 必做功能

### F1. 登录与鉴权

#### 目标

让用户顺利进入移动端系统并保持稳定登录态。

#### 功能点

- 登录页
- 登录成功后默认进入当前会话页
- token 持久化
- token 过期自动回登录
- 退出登录

#### 验收标准

- 用户可在移动浏览器完成登录
- 刷新页面后登录态可恢复
- 登录失效时可正确跳转回登录页

### F2. 当前会话页

#### 目标

作为移动端默认入口和唯一主工作区，承载分析发起、处理中反馈、结果展示和继续追问。

#### 功能点

- 顶部栏
- 左侧抽屉按钮
- 当前会话标题
- 消息流区域
- 欢迎语或空状态引导
- 文本输入框
- 发送按钮
- 快捷提问入口
- 运行中状态条
- 执行摘要卡片
- 完整报告卡片

#### 页面职责

- 新会话时作为欢迎与发起入口
- 会话进行中时展示状态流与后台工作反馈
- 报告生成后承载执行摘要阅读、完整报告入口和后续追问

#### 一期核心规则

- 每个用户同一时间仅允许存在 1 个进行中的分析任务
- 进行中的分析任务包括 `pending`、`processing`、`running`
- 用户可以继续查看历史会话和当前线程，但不能同时发起第二个报告生成任务
- 分析运行中允许继续编辑输入框，但新输入仅保留为草稿，不作为新的有效请求发送到后端

#### 验收标准

- 用户登录后无需再经过首页即可开始分析
- 用户可在同一页面完成“输入 -> 分析 -> 阅读执行摘要 -> 查看完整报告 -> 继续追问”
- 当已有分析任务运行中时，系统能明确阻止第二个分析任务提交

### F3. 会话式分析引导

#### 目标

通过轻量追问补齐分析所需参数，而不是让用户填写复杂表单。

#### 功能点

- 识别股票名称或代码
- 识别基础分析意图
- 信息不足时追加一轮或少量追问
- 使用默认参数减少打断

#### 建议追问方向

- 长期持有还是短期机会
- 更关注收益潜力还是风险控制
- 是否有特定关注点，例如估值、风险、催化

#### 验收标准

- 用户输入一句自然语言即可开始
- 必要时的追问不超过最小必要范围

### F4. 任务状态反馈

#### 目标

让用户在分析处理中获得持续、清晰的阶段反馈。

#### 功能点

- 已接收请求
- 正在识别股票与分析意图
- 正在获取市场与基础数据
- 正在生成多智能体分析结果
- 正在整理结论与报告
- 已完成
- 失败

#### 运行中交互规则

- 任务运行中必须持续显示当前步骤、已用时间、预计剩余时间和最近状态消息
- 状态反馈既要出现在会话消息流中，也要在当前会话页顶部保留轻量状态条
- 若长时间没有阶段切换，仍要通过时间变化和状态文案让用户感知系统仍在工作
- 任务运行中用户可继续输入内容，但点击发送后仅保留草稿，不触发新的分析或追问请求
- 任务运行中应提供 `查看当前任务` 和 `取消任务` 等明确动作入口

#### 验收标准

- 长任务处理过程中有可感知状态变化
- 失败时有明确提示与重试建议
- 用户不会因长时间等待而误以为系统挂死

### F5. 执行摘要卡片

#### 目标

在移动端有限空间内，直接展示现有报告中的执行摘要原文，作为对话内主要阅读内容。

#### 功能点

- 使用报告中的 `执行摘要` 原文
- 默认折叠显示
- 支持展开 / 收起
- 展开后仍保留在当前会话线程中
- 不对现有报告再做二次摘要或模块摘要加工

#### 验收标准

- 用户可在对话线程中快速阅读执行摘要
- 展开后的内容仍然可控，不会把聊天线程变成长文全文页
- 一期不新增对现有报告的摘要加工链路

### F6. 报告详情查看与追问解读

#### 目标

让用户既能查看完整报告，也能继续围绕结果发问。

#### 功能点

- 通过完整报告卡片进入独立阅读层
- 在独立阅读层查看完整报告正文
- 支持长文阅读、返回原会话
- 在原会话中继续追问结果细节
- 支持围绕同一结论继续解释

#### 一期限制

- 先以文本和基础卡片为主
- 暂不追求复杂交互图表
- 不在对话线程中直接展示完整 Markdown 正文
- 不基于现有报告额外生成模块摘要或二次摘要

#### 验收标准

- 用户可顺畅查看完整报告
- 用户可在原线程内继续问“主要风险是什么”“如果我只持有三个月呢”等问题
- 对话线程中的报告相关卡片仅保留“执行摘要卡片”和“完整报告卡片”

### F7. 左侧抽屉与历史会话

#### 目标

让用户以最短路径新建会话、切换历史会话和恢复最近任务。

#### 功能点

- 抽屉顶部品牌区
- `+ 开启新对话`
- 历史会话列表
- 时间分组展示，如今天、昨天、近 7 天
- 点击会话回到对应线程

#### 列表项最小信息

- 会话标题
- 股票标签或任务标签
- 最近更新时间
- 当前状态

#### 验收标准

- 用户能在 3 次点击内回到最近一次会话
- 进行中的任务可被继续查看
- 新建会话动作清晰且高频可见

### F8. 抽屉底部账户菜单

#### 目标

用轻量菜单替代独立“我的页”，满足最基本的账户类操作。

#### 功能点

- 用户头像
- 用户昵称
- 更多按钮
- 系统设置入口
- 帮助与反馈入口
- 退出登录

#### 验收标准

- 账户入口不影响主会话导航
- 用户能快速完成退出登录等基础操作

### 7.2 可做但不是一期必须

- 股票搜索补全
- 最近访问股票
- 任务完成提醒
- 基础消息通知
- 语音输入
- 报告摘要分享

### 7.3 明确不做

- 配置管理
- 平台管理
- 用户管理
- 数据源管理
- 数据库配置
- 重型系统设置页
- 复杂使用统计
- 批量分析
- 复杂股票筛选器
- 完整模拟交易系统
- 学习中心内容体系

## 8. 信息架构

### 8.1 页面清单

一期建议将页面总数压缩到最小：

1. 登录页
2. 当前会话页
3. 完整报告查看页或报告详情层

说明：

- 历史会话通过左侧抽屉直接承载，不再单独强调“历史页”
- “我的”通过抽屉底部账户菜单承载，不再作为主页面

### 8.2 导航结构

推荐导航形态：

- 登录后默认进入当前会话页
- 左上角按钮打开左侧抽屉
- 左侧抽屉承载新对话、历史会话与账户入口

不建议一期使用底部 Tab 导航，原因包括：

- 会与输入框、键盘和发送按钮抢占底部空间
- 会放大“页面并列”的感知，不利于聚焦当前会话
- 不符合单会话工作台的核心定位

### 8.3 左侧抽屉结构

#### 顶部区域

- Logo / 产品名

#### 主操作区

- `+ 开启新对话`

#### 主体区域

- 历史会话列表

#### 底部区域

- 用户头像
- 用户昵称
- 更多按钮

## 9. 关键用户流程

### 9.1 发起分析主流程

1. 用户登录成功
2. 系统进入当前会话页空状态
3. 用户输入“分析宁德时代”
4. 系统识别股票与意图
5. 若信息不足，系统轻量追问
6. 系统展示处理中状态
7. 系统返回执行摘要卡片和完整报告卡片
8. 用户可展开执行摘要原文
9. 用户点击查看完整报告进入独立阅读层
10. 用户返回原会话继续追问“主要风险是什么”

### 9.2 运行中交互流程

1. 用户已提交分析任务
2. 系统在当前会话页展示运行中状态条和状态消息
3. 用户看到当前步骤、已用时间和预计剩余时间
4. 用户可继续在输入框内编写内容
5. 若用户点击发送，系统提示当前分析尚未完成，并将输入保留为草稿
6. 分析任务完成后，草稿可继续发送为正式追问

### 9.3 新建会话流程

1. 用户点击左上角打开抽屉
2. 点击 `+ 开启新对话`
3. 系统创建新会话并进入空白线程
4. 页面显示欢迎语和输入框

### 9.4 历史回看流程

1. 用户打开左侧抽屉
2. 在历史会话列表中选择目标线程
3. 回到原对话内容
4. 查看结果或继续追问

### 9.5 异常恢复流程

1. 用户刷新页面或中断后重新进入系统
2. 系统恢复最近会话与任务状态
3. 若任务仍在进行中，继续显示状态流
4. 若已完成，直接展示执行摘要卡片、完整报告卡片和最新消息

## 10. 会话状态设计

为保证设计、前端和后端协作一致，建议一期至少定义以下会话状态：

1. `idle`
  新会话，等待用户开始输入
2. `collecting_inputs`
  系统正在通过对话补齐分析参数
3. `ready_to_analyze`
  参数已足够，可以开始分析
4. `analyzing`
  任务执行中，展示阶段状态，输入框进入“可编辑但仅保留草稿”的受限状态
5. `report_ready`
  报告已生成，用户可查看执行摘要、打开完整报告并继续追问

如后续需要更细分，可扩展：

1. `report_explaining`
  进入针对报告的追问解释阶段

## 11. 对话式交互规范

### 11.1 输入策略

- 鼓励自然语言输入
- 尽量减少参数选择器
- 优先使用默认配置
- 仅在信息不足时做轻量追问
- 任务运行中允许用户继续编辑输入框，但不接受新的有效发送请求
- 运行中的新输入默认作为草稿保留，待任务完成后再发送

### 11.2 追问原则

- 能自动识别则不追问
- 能合理默认则不追问
- 必须补信息时只追问最关键的一项
- 不做长链路问卷式引导
- 在报告生成完成前，不处理新的正式追问

### 11.3 输出策略

一条完整分析输出建议包含三层：

1. 状态消息
2. 执行摘要卡片
3. 完整报告卡片或详细报告入口

### 11.4 报告展示规则

- 对话线程中的报告相关卡片仅保留两种：执行摘要卡片、完整报告卡片
- 执行摘要卡片直接展示现有报告中的执行摘要原文，并支持折叠 / 展开
- 完整报告卡片点击后进入独立阅读层查看全文
- 一期不基于现有报告再生成模块摘要或二次摘要
- 对话线程不直接承载完整 Markdown 长文正文

### 11.5 会话边界

一期的对话不是开放域助手，而是聚焦以下两类任务：

1. 帮用户完成个股分析输入确认
2. 帮用户理解分析输出结果

## 12. 与现有系统的关系

### 12.1 复用建议

移动端一期建议优先复用：

- 登录鉴权机制
- 后端分析任务 API
- 报告查询 API
- 用户信息 API
- 会话/任务状态接口

### 12.2 不建议直接照搬的内容

- 桌面端后台布局
- Element Plus 管理台交互模式
- 配置管理页面
- 表格密集型页面
- 多级菜单式后台导航

## 13. 成功标准

一期上线后，建议用以下指标评估是否达到目标：

- 用户能成功完成首次分析的比例
- 分析任务完成后的报告查看率
- 用户在同一线程内继续追问的比例
- 历史会话回访率
- 左侧抽屉打开后的会话切换成功率
- 移动端平均单次会话时长
- 运行中任务的中断退出后恢复成功率

## 14. 风险与约束

### 14.1 产品风险

- 若完全做成纯文本聊天，结果会难以阅读
- 若抽屉中只放菜单入口而不直接展示历史列表，会增加切换成本
- 若“我的”入口做得过重，会分散对主会话的注意力
- 若将完整长报告直接塞进聊天线程，会破坏对话节奏并显著增加移动端阅读负担

### 14.2 技术约束

- 需兼容手机浏览器网络波动和页面刷新
- 需确保后端 API 能支持轻量会话式交互体验
- 若任务耗时较长，需要清晰的状态返回机制
- 抽屉切换会话时需要保证线程恢复和滚动定位体验

## 15. 一期开放问题

以下问题建议在下一轮产品讨论中继续确认：

1. 空白会话页是否需要展示快捷问题，还是只保留一句欢迎语？
2. 执行摘要默认折叠展示几行最合适？
3. 历史会话是否需要支持搜索或置顶？
4. `系统设置` 一期是否独立成页，还是先做轻量弹层？
5. 一期是否需要支持分享完整报告或执行摘要？

## 16. 结论

移动端一期建议采用“统一对话页 + 左侧抽屉 + 底部账户菜单”的轻量方案，优先建立以下闭环：

**登录 -> 当前会话 -> 发起分析 -> 查看执行摘要 -> 打开完整报告 -> 继续追问 -> 历史回看**

该方案能够在不复制桌面后台复杂度的前提下，为手机浏览器提供一个真正聚焦、可用、易收敛的第一版产品。
