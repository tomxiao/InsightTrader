# 对话式移动端前端一期 PRD v1

## 1. 文档信息

- 文档名称：对话式移动端前端一期 PRD
- 版本：v1
- 当前方案：统一对话页方案
- 面向对象：产品、前端、后端、设计、测试
- 适用范围：手机浏览器 H5 前端一期

## 2. 背景

当前项目已有一套面向桌面管理台的前端，适合配置管理、后台操作和复杂数据查看，但不适合手机浏览器上的高频使用场景。

移动端一期不追求复制桌面端全部能力，而是聚焦用户最核心的两类任务：

1. 个股分析发起
2. 分析结果解读与继续追问

为降低移动端 UI/UX 复杂度，一期不再采用“首页、线程页、历史页、我的页”并列的传统结构，而是统一为：

**登录后直接进入当前会话页，在一个会话中完成分析发起与结果解读；新建会话、历史切换和账户操作统一收纳到左侧抽屉。**

## 3. 产品定位

对话式移动端前端一期定位为：

**一个面向手机浏览器的轻量投研对话工作台。**

它不是桌面后台的缩小版，而是一个更适合移动场景、更聚焦单任务闭环的使用端产品。

### 3.1 目标用户

- 普通登录用户
- 需要在手机上快速发起分析、查看结论、继续追问的用户
- 已在桌面端完成复杂配置，移动端主要用于“使用”和“消费结果”

### 3.2 一期价值主张

- 用自然语言替代表单式操作
- 用统一会话承载分析过程与结果解释
- 用结构化卡片降低移动端阅读负担
- 用左侧抽屉承载导航，减少主界面干扰

## 4. 目标与非目标

### 4.1 一期目标

- 支持用户在手机浏览器中完成登录、发起分析、查看结果、继续追问
- 建立“登录 -> 当前会话 -> 历史回看 -> 再次追问”的最小闭环
- 降低页面数量与导航复杂度
- 尽量复用现有后端 API、鉴权体系与报告能力

### 4.2 一期非目标

- 不复制桌面端后台管理能力
- 不实现复杂配置管理
- 不实现完整模拟交易终端
- 不实现图表密集型工作台与大表格后台页
- 不将产品做成开放域通用聊天助手
- 不以管理员或运维场景为核心

## 5. 核心设计原则

### 5.1 对话优先

用户优先通过自然语言发起任务，而不是填写复杂表单。

### 5.2 单会话承载主任务

“个股分析”和“报告解读”不是两个页面，而是同一会话中的两个阶段。

### 5.3 少页面、少层级

一期尽量将主体验收敛到一个核心页面，其他能力通过抽屉和轻量菜单承载。

### 5.4 结果卡片化

系统输出不能只有纯文本，必须提供结构化卡片承载关键结论。

### 5.5 默认值优先

能自动识别的参数不要求用户手动选择，必要时仅做轻量追问。

### 5.6 适合中断恢复

手机端用户经常切后台、断网、刷新页面，因此要优先支持任务恢复和历史回看。

## 6. 核心方案概述

### 6.1 登录后默认落点

用户登录成功后，默认直接进入：

- `当前会话页`

当前会话页既承担原本“首页”的欢迎和发起作用，也承担原本“线程页”的处理和解读作用。

### 6.2 左侧抽屉承担的职责

左侧抽屉用于承载辅助导航和会话管理，包括：

- `+ 开启新对话`
- `历史会话列表`
- `底部账户入口`

左侧抽屉不是一个独立业务页面，而是当前会话页的导航容器。

### 6.3 “我的”简化方案

一期不单独做完整“我的页”，而是将其简化为抽屉底部的账户菜单。

账户菜单采用轻量形态：

- 抽屉底部展示头像、昵称、更多按钮
- 点击后弹出轻量菜单

一期建议菜单项仅保留：

- `系统设置`
- `帮助与反馈`
- `退出登录`

## 7. 一期功能范围

### 7.1 必做功能

### F1. 登录与鉴权

#### 目标

让用户顺利进入移动端系统并保持稳定登录态。

#### 功能点

- 登录页
- 登录成功后默认进入当前会话页
- token 持久化
- token 过期自动回登录
- 退出登录

#### 验收标准

- 用户可在移动浏览器完成登录
- 刷新页面后登录态可恢复
- 登录失效时可正确跳转回登录页

### F2. 当前会话页

#### 目标

作为移动端默认入口和唯一主工作区，承载分析发起、处理中反馈、结果展示和继续追问。

#### 功能点

- 顶部栏
- 左侧抽屉按钮
- 当前会话标题
- 消息流区域
- 欢迎语或空状态引导
- 文本输入框
- 发送按钮
- 快捷提问入口
- 运行中状态条
- 研究结果卡片

#### 页面职责

- 新会话时作为欢迎与发起入口
- 会话进行中时展示状态流与后台工作反馈
- 报告生成后承载研究结果阅读和后续追问

#### 一期核心规则

- 每个用户同一时间仅允许存在 1 个进行中的分析任务
- 进行中的分析任务包括 `pending`、`processing`、`running`
- 用户可以继续查看历史会话和当前线程，但不能同时发起第二个报告生成任务
- 分析运行中允许继续编辑输入框，但新输入仅保留为草稿，不作为新的有效请求发送到后端

#### 验收标准

- 用户登录后无需再经过首页即可开始分析
- 用户可在同一页面完成“输入 -> 分析 -> 阅读研究结果 -> 继续追问”
- 当已有分析任务运行中时，系统能明确阻止第二个分析任务提交

### F3. 会话式 Ticker 识别与确认

#### 目标

将用户输入的自然语言转成可供 TradingAgents 执行的标准 ticker。

一期应优先减少用户输入成本，通过少量追问完成确认，而不是要求用户填写结构化表单。

#### 输入与输出契约

输入：

- 用户的原始自然语言问题

输出：

- 可供 TradingAgents 使用的标准 ticker
- 对应股票名称
- 当前识别状态
- 面向前端展示的确认、候选或补充提示

一期建议统一使用以下识别状态：

- `resolved`：已识别出唯一 ticker，可直接进入分析
- `need_confirm`：已有唯一候选，但需要用户确认
- `need_disambiguation`：存在多个候选，需要用户选择
- `not_found`：无法识别，需要用户补充

#### 产品行为约束

- 若已识别出唯一且可信的 ticker，可直接进入分析
- 若已有唯一候选但仍存在不确定性，必须先确认
- 若存在多个可能标的，必须先让用户选择，不得直接发起分析
- 若无法识别，必须明确告知并引导用户补充
- 在正式分析开始前，系统必须明确回显“股票名称（ticker）”
- 整个补充识别流程应尽量控制在 `1~2` 轮内

#### 交互要求

- 识别、确认、候选选择、补充提示都必须在当前会话页完成
- 相关反馈应以消息流的形式呈现，而不是跳转到独立表单页
- 多候选场景优先使用可点击选项，而不是要求用户手输序号
- 在用户完成确认前，不得创建正式分析任务
- 若用户在确认过程中改口，应以最新输入为准，并清空上一轮识别结果

#### 前端交互要求

- 前端不负责用正则或硬编码规则直接决定最终 ticker
- 前端负责承接后端返回的识别状态，并按状态渲染确认、候选或补充提示
- 用户确认完成后，前端再调用正式分析任务创建接口

#### 后端能力要求

后端应基于 LLM Agent 提供独立识别能力，而不是依赖前端直接抽取 ticker。

后端至少应返回：

- 当前识别状态
- 标准 ticker
- 股票名称
- 候选列表（如有）
- 建议回复文案

#### 会话状态建议

识别与确认阶段建议使用以下状态流：

- `idle`：尚未开始
- `collecting_inputs`：正在补充 ticker 信息
- `ready_to_analyze`：已确认股票，可进入分析
- `analyzing`：正式分析中

#### 一期边界

- 一期仅支持单股票识别，不支持一句话中同时分析多个标的
- 一期不支持指数、基金、期货、加密货币，除非后端已明确支持
- 若用户输入的是行业、主题或板块，应明确提示当前仅支持单股票分析
- 若超过 `2` 轮仍无法确定股票，系统应给出清晰兜底提示，而不是继续无限追问

#### 验收标准

- 用户输入明确 ticker 时，系统可直接识别并进入分析
- 用户输入中文股票名时，系统可映射到标准 ticker
- 用户输入模糊简称时，系统能发起确认或候选选择，而不是直接误判
- 存在多个候选时，系统不会直接进入错误分析
- 无法识别时，系统能给出明确补充指引
- 全流程最多 `2` 轮补充即可进入“确认分析”或“无法识别”结论
- 在正式分析开始前，系统始终明确回显“股票名称（ticker）”

### F4. 任务状态反馈

#### 目标

让用户在分析处理中持续感知系统正在工作，知道任务大致进展，并在异常时获得明确提示。

#### 用户可感知能力

- 用户发起分析后，系统会立即反馈任务已开始处理
- 分析过程中，系统会持续展示任务仍在推进，而不是长时间静默
- 用户可以感知任务当前处于“处理中 / 已完成 / 失败”等状态
- 用户可以看到等待时间或阶段文案持续变化，避免误以为系统卡死
- 分析完成后，系统会明确告知结果已可查看
- 分析失败时，系统会明确告知失败并给出下一步提示

#### 交互要求

- 状态反馈必须在当前会话页持续可见
- 状态反馈既要体现在消息流中，也要在页面顶部保留轻量提示
- 状态文案应简洁、稳定、易懂，不暴露内部技术实现细节
- 即使阶段没有明显切换，也应通过时间变化或轻量文案让用户感知任务仍在继续
- 当任务已完成或失败，运行中状态提示应自然退出或切换为结果提示
- 任务运行中，用户仍可继续输入，但新的内容不应立即触发第二个分析任务

#### 一期边界

- 一期只要求用户感知“任务正在推进”，不要求展示底层子任务或内部 Agent 细节
- 一期不要求展示复杂进度条或精确百分比
- 一期不要求展示过于细粒度的技术阶段名称
- 若长时间无结果，系统只需提供清晰提示与重试建议，不要求复杂诊断能力

#### 验收标准

- 用户发起分析后，页面能在短时间内出现明确状态反馈
- 长任务处理中，用户不会因页面静止而误以为系统挂死
- 用户能从界面上区分“处理中 / 已完成 / 失败”三类结果
- 失败时，用户可获得明确提示，而不是无反馈结束
- 整个分析过程中，状态信息始终保持简洁、可理解

### F5. 研究结果卡片

#### 目标

在移动端有限空间内，让用户优先看到最核心的报告结论，并能在当前会话中快速展开阅读。

#### 用户可感知能力

- 报告生成后，用户可直接在会话线程中看到研究结果卡片
- 研究结果默认以较短形态展示，避免长文直接淹没消息流
- 用户可按需展开和收起研究结果
- 展开后，用户仍留在当前会话线程中，不需要跳转到其他页面
- 用户看到的是报告中的核心摘要内容，而不是额外加工出来的新摘要层

#### 交互要求

- 研究结果卡片应作为报告结果的一部分出现在消息流中
- 默认展示长度应足够帮助用户快速判断是否继续深入阅读
- 展开与收起操作必须清晰、低成本
- 展开后的内容应保持可读，但不应破坏聊天线程的浏览体验
- 用户在阅读研究结果后，应能自然进入“继续追问”

#### 一期边界

- 一期仅展示已有报告中的研究结果内容
- 一期不新增二次摘要、模块摘要或个性化摘要能力
- 一期不在对话线程中直接展示完整报告正文
- 一期优先保证可读性与浏览节奏，不追求复杂内容编排

#### 验收标准

- 用户可在对话线程中快速看到核心摘要内容
- 用户可完成展开与收起操作，且交互清晰
- 展开后的摘要仍然可控，不会把聊天线程直接变成长文全文页
- 用户阅读摘要后，能自然进入继续追问

### F6. 分析解读与追问

#### 目标

分析完成后，引导用户围绕自己关注的问题主动提问，系统以各 Agent 的完整分析报告集合为依据进行回答，而非让用户自行阅读原始报告。

#### 用户可感知能力

- 分析完成后，会话线程中展示研究结果卡片，呈现投资总监的最终决策
- 研究结果卡片底部提供引导提示，邀请用户就自己关心的方向继续提问
- 用户可以在同一会话线程中直接提问，如"主要风险是什么"、"适合什么类型投资者"等
- 系统以市场分析、基本面、新闻事件、多空研究、交易计划、风险辩论等完整分析内容为依据回答
- 系统不会把追问当作全新分析任务，而是持续围绕当前分析结果展开解读

#### 交互要求

- 研究结果卡片底部须有明确的引导文案，告知用户可以继续追问
- 追问在原会话线程中完成，无需跳转页面
- 系统回答应有明确的分析依据，而非模糊推断
- 若分析内容不足以支撑某个问题，应明确告知用户无法回答，不得推断或编造

#### 一期边界

- 一期不提供完整报告的独立阅读页，用户无需也无法查看原始报告正文
- 一期不支持跨会话的报告对比或汇总
- 一期不额外生成模块摘要或二次摘要
- 一期重点验证“摘要引导 + 基于报告集合追问回答”的基本链路顺畅可用

#### 验收标准

- 分析完成后，研究结果卡片底部有引导提问的文案
- 用户提问后，系统能基于完整分析报告集合（而非仅靠摘要文本）给出有依据的回答
- 用户围绕同一份分析提问时，系统能保持上下文连续性
- 对话线程中的分析结果卡片仅为研究结果卡片，无完整报告入口

### F7. 左侧抽屉与历史会话

#### 目标

让用户以最短路径新建会话、切换历史会话和恢复最近任务。

#### 用户可感知能力

- 用户可以通过左侧抽屉快速打开历史会话列表
- 用户可以在抽屉中直接新建会话
- 历史会话会按时间分组展示，便于快速定位最近内容
- 用户点击任一历史会话后，可以直接回到对应线程
- 若某个会话仍处于处理中，用户可以重新进入并继续查看当前进展

#### 交互要求

- 抽屉入口必须在当前会话页保持高频可见
- 新建会话动作必须在抽屉中清晰突出
- 历史会话列表应优先帮助用户找回“最近用过的会话”，而不是展示过多次要信息
- 抽屉中的每个会话项都应提供足够信息帮助用户判断是否进入
- 会话切换后，用户应直接回到该线程的当前上下文

#### 一期边界

- 一期只支持基础的历史会话浏览与切换
- 一期不要求复杂筛选、搜索、批量管理或会话编辑能力
- 一期优先保证“快速新建”和“快速返回最近会话”两条主路径顺畅可用

#### 验收标准

- 用户能在 3 次点击内回到最近一次会话
- 进行中的任务可被继续查看
- 新建会话动作清晰且高频可见

### F8. 抽屉底部账户菜单

#### 目标

用轻量菜单替代独立“我的页”，满足最基本的账户类操作。

#### 用户可感知能力

- 用户可以在抽屉底部看到当前登录账号信息
- 用户可以通过底部账户入口打开轻量菜单
- 用户可以在该菜单中完成基础账户操作
- 一期仅保留“退出登录”这一核心操作

#### 交互要求

- 账户入口应固定在抽屉底部，不干扰主会话导航
- 账户入口应足够轻量，不应与历史会话列表竞争主视觉注意力
- 菜单展开与收起应简单直接，用户能明确感知当前状态
- 退出登录操作应易于找到，但不应误触

#### 一期边界

- 一期不单独提供“我的页”或系统设置页
- 一期不承载复杂账户管理、偏好配置或帮助中心能力
- 一期重点验证底部账户入口与退出登录链路是否顺畅

#### 验收标准

- 账户入口不影响主会话导航
- 用户能快速完成退出登录等基础操作

### 7.2 可做但不是一期必须

- 股票搜索补全
- 最近访问股票
- 任务完成提醒
- 基础消息通知
- 语音输入
- 报告摘要分享

### 7.3 明确不做

- 配置管理
- 平台管理
- 用户管理

## 8. 信息架构

### 8.1 页面清单

一期建议将页面总数压缩到最小，仅保留以下核心页面：

1. 登录页
2. 当前会话页

补充说明：

- 历史会话通过左侧抽屉承载，不单独设“历史页”
- 账户相关操作通过抽屉底部账户菜单承载，不单独设“我的页”

### 8.2 导航结构

一期导航结构建议如下：

- 登录后默认进入当前会话页
- 左上角按钮用于打开左侧抽屉
- 左侧抽屉承载新对话、历史会话与账户入口

一期不采用底部 Tab 导航，避免分散用户对当前会话主工作区的注意力。

### 8.3 左侧抽屉结构

左侧抽屉建议保持四段结构：

1. 顶部品牌区
2. 新建会话入口
3. 历史会话列表
4. 底部账户入口
