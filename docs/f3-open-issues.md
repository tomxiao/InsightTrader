## 3. 遗留问题列表

### 3.1 中文别名召回不足，导致部分股票无法命中候选

#### 现象

- 输入 `帮我看看英伟达` 时，系统未识别到 `NVDA`
- 当前评测结果中 `case_4` 返回 `failed`

#### 影响

- 海外股票的中文常用名、音译名、俗称无法稳定命中
- F3 对中文自然语言输入的可用性仍不完整

#### 当前判断

问题主要不在状态机，而在候选召回层。

当前 `StockLookupGateway` 的 alias 主要来自基础资料字段，如：

- `fullname`
- `enname`
- `cnspell`

这类字段不足以覆盖产品态常见中文别名，例如：

- 英伟达 -> `NVDA`
- 苹果 -> `AAPL`
- 谷歌 -> `GOOGL`

虽然 `苹果` 和 `腾讯控股` 当前样本已经能通过，但命中仍依赖模型推断和候选排序共同发挥作用，不代表中文别名层已经完整。

#### 直接影响范围

- `StockLookupGateway`
- F3 中文输入召回稳定性
- 评测样本的中文别名覆盖能力

#### 建议方向

- 为高频股票补一层产品级别名字典或别名映射
- 将 alias 构建从“只依赖基础资料字段”升级为“基础字段 + 业务别名词典”
- 对中文别名命中建立明确回归样本

### 3.2 多轮补全仍存在最终收敛失败

#### 现象

`case_6` 的流程是：

1. 第一轮输入：`重点看估值和护城河`
2. 第二轮输入：`苹果`

当前结果：

- 第一轮返回 `collect_more`
- 第二轮返回 `failed`

#### 影响

- F3 无法稳定完成“先说关注点，再补标的”的多轮会话场景
- 这会直接削弱 PRD 中会话式识别的价值

#### 当前判断

从现有日志看，这次失败不再是规则 fallback 误判，而是 LLM 在第二轮最终结构化输出阶段返回了空内容，导致：

- `resolution_agent_llm_failed error=LLM returned empty structured output`

这说明问题已经收敛到“最终结构化输出链路稳定性不足”，而不是旧版 fallback 的规则误判。

#### 直接影响范围

- `ResolutionAgent` 最终输出阶段
- 多轮上下文承接后的最后一跳收敛
- 评测样本中的上下文补全场景

#### 建议方向

- 对“最终结构化输出为空”增加一次轻量重试
- 在最终输出阶段保留原始响应内容和上下文摘要，便于复盘
- 将多轮补全场景作为必跑回归样本保留

### 3.3 LLM 最终结构化输出仍存在 provider 稳定性风险

#### 现象

当前虽然已经移除 `with_structured_output()`，改成：

- 普通调用
- `response_format={"type": "json_object"}`
- 本地 JSON 解析

但完整评测时依然出现：

- `LLM returned empty structured output`

#### 影响

- F3 的可用性仍受上游 provider 输出稳定性影响
- 同一条链路在代表性样本通过后，完整批量验证仍可能出现偶发失败

#### 当前判断

现阶段的约束已经比之前更强，包含：

- 明确字段名
- JSON Schema
- 合法示例
- `json_object` 输出模式

因此现存问题更像是 provider 运行时稳定性问题，而不是“提示词完全没约束住”。

#### 直接影响范围

- `ResolutionAgent` 的最终输出稳定性
- 脚本批量评测一致性
- 浏览器或真实用户场景中的偶发失败率

#### 建议方向

- 对最终 JSON 输出阶段增加一次重试
- 将空输出、非 JSON 输出、字段缺失输出分开打点
- 在日志中保留更明确的请求轮次、样本 ID、原始响应摘要

建议按以下顺序继续收敛：

1. 先解决最终结构化输出空内容问题

原因：这是当前 `英伟达` 样本失败的核心原因，也决定中文输入的实际可用性。

3. 最后重新执行完整评测集和浏览器走查  
原因：需要确认脚本通过率和产品态表现是否一起收敛。
