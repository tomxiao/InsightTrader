# Backtest Tuning Workflow

`backtest/README.md` 解决的是“怎么跑”。  
这份文档解决的是“怎么高效调优”，避免每轮都重复生成报告、重复看图、重复做无效比较。

## 目标

把回测调优拆成稳定、可复用的工作流：

1. 先验证系统鲁棒性，不先追逐单一标的收益
2. 先识别 `bad_case`，再谈提升盈利能力
3. 每轮实验都保留最小必要产物，方便横向比较
4. 报告生成、信号评价、执行规则评价分开看，避免混淆问题来源

## 总原则

- 不要只看 `summary.json`
- 不要只看单一标的
- 不要在没有 `bad_case` 标签之前直接优化提示词
- 不要同时修改“报告生成逻辑”和“回测执行规则”，否则无法归因
- 每轮只改一个方向，确保前后结果可比

## 推荐调优顺序

### 阶段 1：先做标的扩充

目的不是追求收益，而是确认系统在不同股票上是否稳定。

建议至少覆盖：

- 1 只美股高波动科技股，例如 `AXTI`
- 1 只港股中盘科技股，例如 `1347.HK`
- 后续再补软件股、半导体设备股、震荡股

这个阶段主要回答：

- 系统是否在不同市场里都过度激进
- `sell` 是否普遍偏多
- `hold` 是否经常漏掉大行情
- `buy_on_pullback` 是否只会在某一类股票上有效

### 阶段 2：做信号打标

每个交易日信号都要打标签：

- `good_case`
- `bad_case`
- `unclear`

不要只给成交信号打标签。  
`hold` 和 `sell` 才是后续调优最容易漏掉、但最关键的部分。

最简单可用的打标口径：

- `buy_on_pullback`
  - 已触发且盈利 -> `good_case`
  - 已触发且亏损 -> `bad_case`
  - 未触发或窗口不足 -> `unclear`
- `sell`
  - 信号后 3 个交易日价格明显下跌 -> `good_case`
  - 信号后 3 个交易日价格明显上涨 -> `bad_case`
  - 其余 -> `unclear`
- `hold`
  - 信号后 3 个交易日基本横盘 -> `good_case`
  - 信号后 3 个交易日出现明显单边行情 -> `bad_case`
  - 其余 -> `unclear`

先用简单规则，后面再细化。  
不要一开始就设计太复杂的标签体系。

### 阶段 3：做 bad case 归因

每轮调优的重点不是“多赚了多少”，而是先回答：

- 哪类信号最容易错
- 错在报告内容、信号解析、还是执行规则

建议把 `bad_case` 归到固定类型里：

- 过早 `sell`
- 过晚 `sell`
- 应该 `buy` 却给了 `hold`
- 应该 `hold` 却给了 `sell`
- 支撑区过宽，导致过早触发
- 失效价过松，导致止损过晚
- 宏观风险权重过高
- 个股趋势权重过高

只有归因稳定后，再改提示词或规则。

### 阶段 4：做小步调优

每轮只允许一种调优动作：

- 只改决策摘要提示词
- 或只改 `report_parser.py`
- 或只改 `execution_rules.py`

不要在同一轮里同时改：

- 分析提示词
- `buy/hold/sell` 解析规则
- 止损或退出逻辑

否则你只能看到“结果变了”，看不出“为什么变了”。

### 阶段 5：做回归验证

每次调优后至少做两类验证：

- 原有标的回归，例如 `AXTI`
- 新标的抽样，例如 `1347.HK`

如果一个改动只改善了 `AXTI`，但让港股样本明显变差，就先不要继续放大。

## 单轮实验标准流程

每轮实验按下面顺序走：

1. 固定本轮目标
2. 固定样本池
3. 生成历史报告
4. 跑回测
5. 给每天信号打标签
6. 汇总 `bad_case`
7. 只改一类问题
8. 重新跑同一批样本
9. 和上一轮做横向对比

### 1. 固定本轮目标

一轮实验只能有一个主目标，例如：

- 减少“过早 sell”
- 减少“hold 漏掉上涨”
- 提高 `buy_on_pullback` 的触发后胜率

不要写“提升整体效果”这种模糊目标。

### 2. 固定样本池

建议为每轮实验固定一个最小池：

- `AXTI`
- `1347.HK`

等这两只流程稳定后，再扩成 4 到 8 只。

### 3. 生成历史报告

按 [README.md](/D:/CodeBase/InsightTrader/backtest/README.md) 里的流程生成：

- `batch_metadata.json`
- `report_manifest.csv`
- `reports/`

同一轮尽量保持：

- 相同日期区间
- 相同频次
- 相同模型

### 4. 跑回测

至少保留：

- `signals.csv`
- `trades.csv`
- `summary.json`
- `*-ohlcv.csv`
- `*-bt.png`

### 5. 给每天信号打标签

建议每个批次目录都补一份：

- `signal_case_labels.md`

这份文件应包含：

- 标签口径
- 每日标签
- 需要优先复盘的 `bad_case`

### 6. 汇总 bad case

不要直接看所有样本。  
先按类型聚合，例如：

- 连续 `sell` 后股价反而反弹
- `hold` 后股价连续下跌
- 唯一的 `buy` 触发后是否合理

### 7. 只改一类问题

示例：

- 如果 bad case 主要是“反弹初期仍连续 `sell`”，优先调低宏观风险在反弹场景下的权重
- 如果 bad case 主要是“下跌段还在 `hold`”，优先提升趋势破坏时的 `sell` 敏感度

### 8. 重新跑同一批样本

必须先回归同一批样本，再换新样本。  
否则你分不清变化来自：

- 参数改动
- 标的差异
- 市场差异

### 9. 和上一轮做横向对比

每轮至少比这几项：

- `signal_count`
- `buy/hold/sell` 分布
- `trade_count`
- `triggered_trade_count`
- `win_rate`
- `avg_return`
- `bad_case` 数量
- `good_case / bad_case / unclear` 比例

## 推荐产物规范

建议每一轮实验都在对应输出目录保留：

- `summary.json`
- `signals.csv`
- `trades.csv`
- `signal_case_labels.md`
- `notes.md`

其中 `notes.md` 建议固定写这几项：

- 本轮目标
- 本轮改动
- 观察到的主要变化
- 新增 bad case
- 消失的 bad case
- 是否进入下一轮

## 推荐节奏

### 先做周节奏，不要每天改

如果你开始同时做多个标的，不建议边看边改。  
更有效的节奏是：

1. 一次性跑完一批样本
2. 一次性打标签
3. 一次性归类 bad case
4. 只做一轮集中修改
5. 再统一回归

### 每轮结束必须回答 3 个问题

1. 本轮主要减少了哪类 bad case
2. 本轮是否引入了新的坏模式
3. 本轮改动是否能跨标的成立

如果这 3 个问题答不清，就不要进入下一轮优化。

## 什么时候进入“提升盈利能力”阶段

只有同时满足下面条件，才建议从“减少 bad case”转入“提升盈利能力”：

- 至少 2 个标的上信号分布没有明显失控
- `bad_case` 已能归成少数几类固定模式
- `sell` 和 `hold` 的判断不再经常完全错向
- 回测结果对单一标的不再高度依赖

否则继续优化盈利，大概率是在放大过拟合。

## 当前建议

基于现在的进度，建议下一步固定使用下面的最小工作流：

1. `AXTI` 和 `1347.HK` 作为第一批双市场样本
2. 每轮都输出 `signal_case_labels.md`
3. 优先减少以下两类 bad case

- 反弹初期仍连续 `sell`
- 下跌延续阶段错误给 `hold`

先把这两类问题压下去，再进入下一阶段。
